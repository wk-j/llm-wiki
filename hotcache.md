- Ingested **"Loop Engineering" — Addy Osmani** (x.com, 2026-06-09). Created `[[loop-engineering-osmani]]`, `[[loop-engineering]]`, `[[git-worktrees]]`. Updated `[[addy-osmani]]`, `[[orchestration-tax]]`, `[[cognitive-surrender]]`, `[[subagent-patterns]]`, `[[long-running-agents]]`. Key point: **loop engineering = stop prompting the agent yourself, design the system that prompts it** (Steinberger: "design loops that prompt your agents"; Cherny: "my job is to write loops"). A loop = a [[agent-harness-engineering|harness]] **one floor up** — it runs on a timer, spawns helpers, feeds itself; a *recursive goal* (set a purpose, AI iterates until done). Needs **five pieces + memory**: (1) **Automations** = the heartbeat (schedule/discovery/triage; `/loop` re-runs on cadence, `/goal` runs across turns until a *verifiable* stop condition that a **separate small model** checks — maker≠checker); (2) **Worktrees** = parallel agents don't collide on files (`--worktree`, `isolation: worktree`); (3) **Skills** = project intent written on disk (skill = authoring format, plugin = shipping); (4) **Plugins & Connectors** = MCP into real tools (open PR, link Linear, ping Slack); (5) **Sub-agents** = maker/checker split (the writer is too nice grading itself); (6) **state file** = memory outside the single conversation ("the agent forgets, the repo doesn't"). **Claude Code & Codex both have all five now** → design the loop, not the tool. Three problems get *sharper* as the loop improves: verification still yours ("done is a claim, not a proof"), **comprehension debt** (cousin of [[cognitive-debt]]), **[[cognitive-surrender]]**. Worktrees remove mechanical collision but **YOU are still the ceiling** (review bandwidth) → ties straight into [[orchestration-tax]]. "Build the loop. Stay the engineer."
- Ingested **"Harness engineering for coding agent users" — Birgitta Böckeler / Thoughtworks** (martinfowler.com, 2026-04-02). Created `[[harness-engineering-bockeler]]`, `[[birgitta-bockeler]]`, `[[thoughtworks]]`, `[[harness-guides-sensors]]`. Updated `[[coding-harness]]`, `[[harness-engineering]]`, `[[harness-ratchet]]`, `[[shift-left-testing]]`. Key point: a mental model for the coding-agent **user** harness (vs the builder harness baked into the tool). Harness = **guides (feedforward)** + **sensors (feedback)**, each split **computational** (CPU, deterministic, fast, cheap → run every change) vs **inferential** (model/LLM-judge, semantic, slow/expensive/non-deterministic → run selectively). Sensors feed back into the agent's *self-correcting loop*, not just humans → less review toil. The **steering loop**: humans iterate the harness; recurring failures become improved controls (= [[harness-ratchet]]). **Keep quality left** (shift-left): distribute sensors by cost/speed across the lifecycle. Harness = **cybernetic governor** with 3 **regulation categories**: maintainability harness (easiest, tooling exists), architecture fitness harness (= fitness functions), **behaviour harness** (the unsolved "elephant" — over-relies on AI-generated tests; approved-fixtures helps selectively). **Harnessability** varies (typed langs / clear module boundaries / frameworks help; legacy paradox: harness most needed where hardest to build). **Harness templates** as a future direction. Field evidence: OpenAI harness write-up, Stripe minions, Thoughtworks "janitor army". `[[fitness-function]]` referenced but not yet created.

- Ingested **"The Orchestration Tax" — Addy Osmani** (X post, 2026-05-28). Created `[[the-orchestration-tax]]`, `[[orchestration-tax]]`, `[[cognitive-surrender]]`. Updated `[[addy-osmani]]`, `[[theory-of-constraints]]`, `[[local-optimization-trap]]`, `[[acceptance-bottleneck]]`, `[[care-allocation]]`, `[[cognitive-debt]]`, `[[subagent-patterns]]`, `[[dynamic-workflows]]`, `[[long-running-agents]]`. Key point: spawning agents is cheap but **closing the loop (review + merge) routes through one serial human**; you are the **GIL** of your agent fleet and **Amdahl's Law** caps speedup at the serial fraction (= judgement). Optimizing the non-bottleneck just deepens the queue before review. Fix = **architect your attention**: scale fleet to review rate (backpressure), sort work (never parallelize judgement-heavy), batch reviews, spend the lock only on judgement (agents self-verify the boring 80%), protect serial time. **Busy ≠ productive**; unpaid tax = technical + cognitive debt. This is the **consumption-side counterweight** to production-side pages (subagent-patterns, dynamic-workflows, agent-swarm). Name coined by Richard Seroter on a Google I/O panel.
