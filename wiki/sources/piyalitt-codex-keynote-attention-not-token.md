# Piyalitt Ittichaiwong — Attention ไม่ใช่ Token / Codex Keynote

โพสต์ Facebook ของ [[piyalitt-ittichaiwong|Piyalitt Ittichaiwong]] สรุป keynote ของทีม [[openai-codex|Codex]] ในงาน [[ai-engineer-worlds-fair|AI Engineer World's Fair]] งานนี้รวม AI engineer กว่า 7,000 คนไว้ในที่เดียว ช่วงหลักนำโดย [[romain-huet|Romain Huet]] (หัวหน้า Developer Experience ของ OpenAI) กับ [[alexander-embiricos|Alexander Embiricos]] (หัวหน้าฝ่ายผลิตภัณฑ์ Codex) แล้วส่งต่อให้ [[peter-steinberger|Peter Steinberger]] (the Clawfather — ผู้สร้าง OpenClaw ที่ตอนนี้ร่วมงานกับ OpenAI)

โพสต์เปิดด้วยสรุปสั้น ๆ ของ [[thariq-shihipar|Thariq]] จากทีม Claude Code เรื่อง map vs territory และ [[unknowns-matrix|unknowns 4 ช่อง]] (มีหน้าเต็มใน [[field-guide-to-fable-finding-unknowns]]) แล้วลงลึก keynote หลัก

## แก่นความคิด

พอ model เก่งขึ้น **คอขวดก็ย้ายจาก AI มาอยู่ที่มนุษย์** เรื่องสำคัญไม่ใช่แค่ prompt สวย แต่คือเรารู้ทัน unknowns แค่ไหน และใช้ [[attention-bottleneck|attention]] กับจังหวะที่คุ้มจริงหรือเปล่า. ฝั่ง OpenAI/Codex ยืนยันว่า engineering ไม่ได้ตาย แค่งานของ engineer เปลี่ยนไป คนที่ได้เปรียบคือคนออกแบบ [[inner-loop-outer-loop|outer loop]] ให้ดี แล้วปล่อยให้ agent ทำงานวนชั้นในแทน

> "ตอนนี้สิ่งที่จำกัดผมมากที่สุดไม่ใช่ token หรือ compute อีกต่อไป แต่คือ attention และมันเพิ่มไม่ได้"
> — Peter Steinberger บนเวที (อ้างในโพสต์)

> "อนาคตไม่ใช่การเปิด terminal 20 หน้าต่าง แต่คือ loop ที่ดีกว่าเดิม มาสร้างมันกันเถอะ"
> — Peter Steinberger ปิดท้าย keynote

## ส่วนเปิด: Thariq กับ unknowns (สรุปย่อ)

Piyalitt เปิดโพสต์ด้วยกรอบคิดจาก Thariq ซึ่งต่อกับหน้าเดิมใน wiki ได้ตรงมาก:

- **"The map is not the territory"** — prompt/context คือแผนที่; codebase/โลกจริงคือพื้นที่จริง; ช่องว่างคือ unknowns ที่ AI ต้องเดา
- ยิ่ง model เก่ง คอขวดยิ่งไม่ใช่ตัว AI แต่เป็นความสามารถของเราที่จะทำเรื่องคลุมเครือให้ชัดก่อนลงมือ
- คนเก่ง agentic coding ไม่ใช่คนพิมพ์ prompt สวย แต่คือคนที่ลด unknowns ได้เก่ง และรู้ว่า unknowns แบบไหนต้องจัดการก่อน
- เทคนิค 3 จังหวะ: ก่อนเขียน (blind spot pass, brainstorm/prototype, interview), ระหว่างเขียน (`implementation-notes`), หลังเสร็จ (quiz ก่อน merge)
- ตัวอย่าง color grading ในวิดีโอ launch Fable: ไม่สั่ง AI ทำมั่ว แต่ให้สอนก่อนเพื่อให้รู้ว่า "ดี" หน้าตาเป็นยังไง

**ผลคือ:** explainer, brainstorm, interview, และ prototype ไม่ใช่พิธีกรรมเพิ่มงาน แต่เป็นวิธีราคาถูกในการหา blind spot ก่อนที่มันจะแพงเกินแก้ — ตรงกับประโยคใน [[unknowns-matrix]]

## งาน World's Fair ของชาว AI engineer

[[romain-huet|Romain]] เปรียบงานนี้กับ World's Fair ในอดีต ภาพที่เขาอยากให้เห็นคืออนาคตไม่ได้ลอยมาจากไหน แต่มันถูกสร้างต่อหน้าคนทั้งห้อง. ในวันที่หลายคนถามว่า engineer จะหายไปไหมเมื่อ code กลายเป็นของนามธรรม ทีม Codex ตอบกลับอีกทาง:

- Software เคยกินโลก → AI กิน software → **AI engineer กำลังกินโลก**
- แก่น engineering ไม่เคยเป็นการพิมพ์ code แต่คือแก้ปัญหาด้วยวิทยาศาสตร์ + การออกแบบ + รสนิยม + วิจารณญาณ + จินตนาการ
- ยุค AI = กลับสู่รากเหง้า ไม่ใช่จุดจบ

## จาก 15 เดือน เหลือ 6 สัปดาห์

ตัวเลขที่สะท้อนความเร็วของวงการคือ OpenAI เคยปล่อย model ใหม่ประมาณทุก 15 เดือน แต่ตอนนี้จังหวะสั้นลงเหลือราว 6 สัปดาห์. ก่อนงานไม่นานก็เพิ่ง preview ตระกูล GPT-5.6

ประสบการณ์ coding วิวัฒน์เป็นขั้นบันได:

1. autocomplete ธรรมดา
2. inline prediction
3. Cmd+K (แก้ code ได้แต่ยังไม่ตรวจงานตัวเอง)
4. model รัน test ตรวจงานตัวเอง
5. วันนี้: model รับเป้าหมายยาวและทำต่อเนื่องจนจบ

**ตัวอย่างบนเวที:** DevDay 2024 Romain ใช้ o1-preview สร้างหน้าจอควบคุม drone แบบสด — โมเดลยังรัน/ตรวจงานตัวเองไม่ได้ เลยลุ้นไปกับมัน (โอกาส demo รอด ~3 ใน 4 → ฉายา "the demo god"). DevDay 2025 แค่ปีเดียวถัดมา ปล่อยให้ agent คุมกล้องและไฟทั้งงานได้

## Agent ก่อนและหลังการเขียน code

ปี 2026 ทีม Codex ปล่อยของต่อเนื่อง. Alexander ยกสามอย่างที่เขาชอบเป็นพิเศษ: **Codex app**, **Goal Mode**, **Remote**. ทั้งหมดนี้เกิดจากทีมใช้ Codex สร้าง Codex เอง

จุดเปลี่ยนคือ agent เริ่มทำงานบนคอมพิวเตอร์ได้แทบทุกอย่างแล้ว มันไม่ได้ช่วยแค่ตอนเขียน code แต่ช่วยตั้งแต่ก่อนเขียน (ทำไมงานนี้ต้องเกิด) ไปจนถึงหลังเขียน (review, deploy). ยิ่ง agent เข้าใจเหตุผลของงานและต่อกับขั้น review/deploy ได้มากเท่าไหร่ มันก็ยิ่งเริ่มงานเองและพาไปถึงเส้นชัยได้มากขึ้น

ในมุม product ของ Alexander สิ่งที่น่าตื่นเต้นไม่ใช่แค่ความเร็ว แต่คือ **ตัดสินใจได้ดีขึ้นว่าควรสร้างอะไร**. เพราะ prototype ได้หลายไอเดียกว่าเดิม ทีมเลยมีเวลาคลุกกับ user จริงมากขึ้น

## เป้าหมายไม่ใช่แทนที่ engineer

ทีมยอมรับตรง ๆ ว่างานคอมพิวเตอร์ความยาวปานกลาง ถ้าให้คนกับ model ใช้เวลาเท่ากัน โดยเฉลี่ย model น่าจะทำได้ดีกว่า. แต่คำตอบของทีมไม่ใช่ “แทนที่ engineer” คำตอบคือ **เสริมพลัง engineer**

ภาพเปรียบเทียบคือเวลาทำงานกับทีมคนจริง เรามักคุยเป้าหมายแล้วปล่อยให้ทีมลงมือ ไม่มีใครยืนมองข้ามไหล่ทุกชิ้น แต่บางจังหวะเราก็อยากลงลึก. หลักออกแบบจึงเป็นว่า **ผู้ใช้ต้องรู้สึกเชี่ยวชาญและเป็นเจ้าของงานเสมอ** ถ้าอยากเข้าไปดูรายละเอียด ต้องเข้าได้ง่าย

### เบื้องหลัง Codex app ที่เคยโดนค้าน

ตอน pitch มี developer บอกว่าไม่มีวันออกจาก terminal/Vim/Emacs. วันนี้คนกลุ่มเดียวกันกลับกลายเป็นผู้ใช้. เหตุผลของทีมคือ:

- **CLI** ไม่ใช่หน้าจอทำงานร่วมกันที่เหมาะกับงานทุกประเภท — ธรรมชาติของมันยังเป็น chat เป็นหลัก
- **IDE** บังคับให้เริ่มจาก code ทั้งที่งานยุคนี้มักควรเริ่มจากคุยให้เข้าใจก่อน แล้วค่อยลงลึก
- **Codex app** คือ chat ที่เรียบง่าย ใช้ได้ทั้ง coding และงานอื่น — ชี้จุดแล้วสั่งแก้เฉพาะตรงนั้นได้ หรือลงมือแก้เองก็ได้

Alexander บอกว่าเขาอ่านนิยายวิทยาศาสตร์เยอะ และมองว่า chat ถูกประเมินค่าต่ำเกินไป. โลกต้องการตัวช่วยที่คุยด้วยได้เรื่องเดียวกันทุกที่ และต้องมีหน้าจอทำงานร่วมกันที่ทรงพลังพอสำหรับตอนที่เราอยากตรวจ ปรับทิศ หรือปั้นงานด้วยมือ

## Stack ที่เปิดทุกชั้น

ชั้น harness ของ Codex **open source ทั้งหมด** จึง fork หรือดัดแปลงได้. ทีมใช้ชื่อกลาง `AGENTS.md` แทนการประดิษฐ์รูปแบบใหม่ เพื่อให้ agent ค่ายอื่นใช้ร่วมกันได้ด้วย. model ของ OpenAI เป็นค่าเริ่มต้น แต่ไม่ได้ล็อกไว้กับ OpenAI เท่านั้น ผู้ใช้สลับ open model เข้าไปใน agent loop เดิมได้. ที่สำคัญคือ harness ชุดเดียวกันนี้ถูกใช้ตอน post-training ด้วย model จึงหัดเรียก tool ในสภาพแวดล้อมเดียวกับที่ developer ใช้จริง

ตัวอย่างคือ Thomas/Dimillian บน X ใช้ App Server สร้าง **Codex Monitor** ก่อน Codex app เปิดตัว วันนี้เขาเข้ามาร่วมทีมและสร้าง Codex สำหรับ iOS. ชั้นแอปยังขยายต่อได้อีก ทั้ง browser ในตัว, plugin system, browser use และ computer use ที่เสียบผ่านจุดต่อเดียวกับที่เปิดให้ทุกคน. มี plugin สาย data science และ design ที่เปิด open source ด้วย. subscription เดิมยังใช้ได้ใน OpenCode, Pi, Droid, OpenClaw, Xcode, JetBrains

**สรุปชั้นนี้:** ไม่ได้สร้างระบบหนึ่งใช้เองแล้วสร้างอีกระบบลดทอนให้ developer — ทุกชั้นใช้ของชิ้นเดียวกับที่ส่งมอบ

## [[value-maxing|Value maxing]] — ฉลาดขึ้น ถูกลง เร็วขึ้นพร้อมกัน

ทีมเรียกแนวคิดนี้ว่า **value maxing** ไม่ใช่ token maxing. เวลาคุยกับผู้บริหาร engineering คำถามมักวนอยู่ตรงนี้: ทำอย่างไรให้ได้ความฉลาดมากขึ้น ในราคาที่คุมได้ และเร็วพอจะเปิด workflow ใหม่

**แกนคุ้มค่า:**

- ทุกคนอยากได้ความฉลาดระดับ frontier + ปริมาณความฉลาดต่อเงินสูงสุด
- ตัวอย่าง: GPT-5.6 **Terra** — ความฉลาดระดับ GPT-5.5 ในราคาครึ่งเดียว
- **Luna** — ชนะ model ดังหลายตัว ราคา $1/1M input, $6/1M output
- รุ่นบนสุดตระกูล 5.6 ขึ้นนำ Terminal Bench (ตามกราฟบนเวที — ยัง source-attributed)

**แกนความเร็ว:**

- GPT-5.3 Codex Spark พิสูจน์แล้วว่าความเร็วปลดล็อกอะไรได้ แต่คนไม่อยากแลกความฉลาด
- รอบนี้: frontier ของตระกูล 5.6 รันบน Cerebras ที่ **750 token/วินาที** — เปิดใช้เดือนถัดไป
- ภาพเปรียบเทียบคือได้ PR ขนาดใหญ่ในราว 10 วินาที
- ประเด็นไม่ใช่แค่คำตอบเดียวเร็วขึ้น แต่คือ agent สามารถลอง 5–6 แนวทางขนาน แล้วคัดอันที่ดีที่สุดได้ในเวลาน้อยกว่าที่เมื่อก่อนใช้สร้างคำตอบเดียว

## ปิดโน้ตบุ๊กแล้วปล่อย agent ทำงานต่อ

ตอนถามคนในห้องว่าใครเคยเปิดโน้ตบุ๊กทิ้งไว้ในออฟฟิศเพื่อให้ agent รันต่อ มือยกกันทั่วห้อง. สิ่งที่ทุกคนอยากได้คือปิดเครื่องตัวเองได้ แต่งานหลายชิ้นยังรันขนานต่อบนเครื่องของมันเอง และแยกกันชัดเจน

การเปิดตัวใหญ่ครั้งแรกของ Codex คือ **Codex Cloud** และกำลังมีอัปเกรดสำคัญตามมา. ภาพอนาคตคือ **ไม่มีเส้นแบ่งแปลก ๆ ระหว่าง local กับ cloud** อีกต่อไป เราคุยกับ agent ตัวเดียวได้ทุกที่ แล้วมันเลือกเองว่างานนี้ควรใช้สภาพแวดล้อมแบบไหน

มีการอ้างทำนายของ Theo บนจอ — ผู้พูดฟันธงว่าภาพแบบนั้นจะมาถึงเร็วกว่า 6 เดือน

## บทเรียนจาก Peter: orchestrate หรือแค่ polling?

Peter เปิดด้วยภาพเมื่อเดือนมกราคม เขาเปิด terminal พร้อมกันเกิน 10 หน้าต่าง แล้วคอยเฝ้าว่าอันไหนเสร็จเพื่อป้อนงานใหม่. ตอนนั้นรู้สึกเหมือน productivity ขั้นสุด แต่พอมองย้อนกลับ วันนี้มันออกจะดูตลก

ประโยคคมคือ **นึกว่ากำลัง orchestrate แต่จริง ๆ แค่ polling**. ตัวเขาเองเป็น scheduler, router, และ memory ของทั้งระบบ. พอขยับจาก pair กับ agent ทีละตัวไปเป็น 10 terminal งานจึงไม่ใช่ coding อย่างเดียวแล้ว แต่กลายเป็นการบริหารลูกทีม 10 คน

วิธีทำงานวันนี้เปลี่ยนเป็นคุยกับ **manager agent ตัวเดียว** ที่รันต่อเนื่องระยะยาว แล้วให้มันกระจายงานไป worker. งานไหนยากจริงค่อยลงไป pair กับ worker โดยตรง. เขาสรุปภาพนี้ว่า *"ผมบริหารผู้จัดการของบริษัทเล็ก ๆ ที่พนักงานเป็น agent"*

สามอย่างที่ทำให้ภาพนี้เป็นจริง:

1. **Server-side compaction** — งานรันยาวเสถียร ไม่ต้องเปิด session ใหม่เลี่ยงปัญหา
2. **ระบบประสานงาน** — thread เดียวสร้างและคุมหลาย project
3. **Automation** — ปลุก manager ตัวเดิมเมื่อมีเหตุการณ์

สามอย่างนี้รวมกันเป็น context ที่คงอยู่ การกระจายงาน และตัวกระตุ้นให้ระบบตื่นขึ้นมาทำงานต่อ — **นี่คือ loop ของเรา**

## Bottleneck ย้ายที่ — จาก token → compute → attention

ปีที่แล้วข้อจำกัดหลักของ Peter คือ token เขาแก้ด้วยการเข้าไปทำงานที่ OpenAI และแซวตัวเองว่าวิธีนี้ scale ไม่ได้. จากนั้นคอขวดย้ายไปที่ compute เพราะ thread รันพร้อมกันจน MacBook ดังเหมือนเครื่องยนต์เจ็ต เขาแก้เกือบหมดด้วย **test box** ให้ agent รัน test บนเครื่องอื่น. **ข้อจำกัดวันนี้คือ attention** และต่างจาก token/compute ตรงที่เพิ่มไม่ได้

ทักษะสำคัญจึงกลายเป็นการเลือกว่าจะใช้ attention ไปกับอะไร. Peter ถามบนเวทีว่าเรายังนั่งจ้อง agent ระหว่าง code ไหลผ่านหน้าจออยู่ไหม? กับ model รุ่นก่อน พฤติกรรมนี้จำเป็น เพราะถ้าเห็น agent เลี้ยวผิดต้องรีบกด escape ดึงกลับ. แต่ model รุ่นล่าสุดเข้าใจเจตนาได้ดีขึ้นมาก การนั่งดู code ถูกเขียนจึงเริ่มกลายเป็นการเผา attention เปล่า ๆ

## Inner loop / outer loop

Peter วาด loop ที่เขาใช้จริงกับ open source project:

1. มีคนเปิด issue เข้ามา
2. **Manager** ตื่นขึ้นมา → เทียบ issue กับเป้าหมาย/วิสัยทัศน์ของ project → ตัดสินใจว่าน่าทำไหม
3. ถ้าใช่ → สร้าง **worker** หนึ่งตัวให้สืบหา แก้ และรัน test
4. **Agent อีกตัว** รีวิวผลลัพธ์ซ้ำอีกชั้น
5. Peter ไม่ต้องดู agent ทำงานหรืออ่านทุกข้อความ — เมื่อ manager ต้องการเขาจริง ๆ จะส่ง PR พร้อม issue ต้นทาง + diff (บางครั้งแถมวิดีโอ/build ให้ VNC ดู)
6. Peter รีวิวรอบเดียว ทิ้งโน้ต หรืออนุมัติ → loop หมุนต่อจน merge เองหลังผ่านการตรวจครบ

หลักการคือ **agent ทำงานวนชั้นใน ส่วนมนุษย์กำหนดทิศทางและตัดสินใจในชั้นนอก**

## โจทย์ที่ยังไม่มีใครแก้ได้

พอ manager agent มีอายุยืนขึ้น การผูกมันไว้กับโน้ตบุ๊กเครื่องเดียวก็เริ่มผิดฝาผิดตัว. Codex ย้ายงานข้ามเครื่องได้แล้ว และ OpenClaw มี gateway/node แต่ Peter บอกว่ายังไม่มีอันไหนเป็นรูปแบบสุดท้าย. ภาพที่เขาอยากได้คือ agent เชื่อมทุกเครื่องของเขาเอง และรู้เองว่างานไหนควรไป cloud งานไหนต้องอยู่ local

Manager ที่แท้จริงไม่ควรเป็นแค่ session ในแอปเดียว ควรเป็น agent ที่ส่งข้อความหาได้ สั่งผ่าน Slack ได้ และรายงานได้ไม่ว่าเราอยู่ที่ไหน. คำถามทิ้งท้ายคือ **ทำไมเรายังแค่คุยกับ agent แล้วให้มันออกแบบ loop ทั้งหมดให้เราเองไม่ได้?** คำตอบวันนี้คือยังไม่มีใครแก้สำเร็จ

Peter ปิดว่า model พัฒนาเร็วกว่า harness และองค์กรที่ห่อหุ้ม — การออกแบบสิ่งเหล่านี้คือโจทย์ engineering ข้อถัดไป

## มุมมองส่งท้ายจากเพจ

- อาชีพ engineer ถูกนิยามใหม่ ไม่ใช่ถูกลบ — ขยับจากคนพิมพ์ code ไปเป็นคนออกแบบระบบ กำหนดทิศทาง และตัดสินใจใน outer loop
- ทรัพยากรแพงที่สุดคือ attention — token ซื้อเพิ่มได้ compute เช่าเพิ่มได้ แต่สมาธิมีเท่าเดิม
- ระบบที่ดีควรรบกวนเราเฉพาะจังหวะที่การตัดสินใจของมนุษย์สร้างความแตกต่างได้จริง
- stack ที่เปิดกว้างมีแนวโน้มชนะ — OpenAI ใช้ stack เดียวกับที่ส่งให้ developer ส่งสัญญาณว่าอนาคต agent ถูกสร้างร่วมกันทั้ง ecosystem

## Key takeaways

1. **คอขวดย้ายที่** — จาก model → unknowns/attention ของมนุษย์; token/compute เพิ่มได้ แต่ attention เพิ่มไม่ได้
2. **Engineering ไม่ตาย** — งานขยับกลับไปที่แก่นเดิม: แก้ปัญหา ออกแบบระบบ และตัดสินใจใน outer loop
3. **Codex stack เปิดทุกชั้น** — harness, AGENTS.md, App Server, plugin; ใช้ของชิ้นเดียวกับที่ส่งให้ developer
4. **Value maxing** — ฉลาด + ถูก + เร็วพร้อมกัน (Terra, Luna, Cerebras 750 t/s) ไม่ใช่แค่ token maxing
5. **อนาคต = loop ไม่ใช่ 20 terminal** — manager agent + worker + compaction + automation; มนุษย์ไม่ polling แต่ตัดสินใจ
6. **โจทย์เปิด:** agent ที่ออกแบบ loop เองทั้งหมด + manager ที่ไม่ผูกกับเครื่อง/แอปเดียว — ยังไม่มีใครแก้สำเร็จ
7. **เชื่อมกับ Thariq:** ทักษะที่คุ้มลงทุนคือรู้จักสิ่งที่ยังไม่รู้ ไม่ใช่แค่สั่งให้ model เก่งขึ้น
