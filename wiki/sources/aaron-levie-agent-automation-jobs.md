# Aaron Levie — Agent automation requires dedicated people

X post by **[[aaron-levie|Aaron Levie]]** (CEO of Box), captured 2026-04-24. Argues that the gap between "we have a chatbot" and "we have an agent automating a real process" is large enough that companies will need **dedicated people** — not side-project volunteers — to close it.

## Core claim

> The jump from working with a chatbot to having an agent that actually helps automate a process requires a real amount of work.

The work is too technical and too cross-cutting to be left to "every individual employee" as a side project. Companies need a named role that owns it.

## The job spec (as listed)

A single role expected to cover all of:

1. **Map out new workflows** with agents — identify which existing processes are candidates and how an agent should fit in.
2. **Implement new systems to deploy agents** — production hosting, not POCs.
3. **Keep agent context fresh** — make sure the agent has the right, up-to-date information.
4. **Wire internal systems** to connect to the agents — integration plumbing.
5. **Create evals** for the agents — measurable quality bars.
6. **Figure out where the human is in the loop** — HITL design.
7. **Manage the system when there are new upgrades** — model swaps, version drift.
8. **Help with change management** of the existing business process — the org-side of the deployment.

This is a superset of "AI engineer" — it includes both technical plumbing and business-process change management.

## Where it sits in the org

Three plausible homes:

- **IT** — closest to existing systems integration and deployment muscle.
- **Engineering** — closest to the eval / context / model-swap craft.
- **Inside the business function itself** — closest to workflow knowledge and change management.

Levie's prediction: this becomes the **future of software engineering inside non-tech companies** — a huge growth area, even as the headcount profile inside tech companies stays bounded.

> "Most companies will have to be hiring for this now or in the future, and it's another example of the kind of new jobs that will be created in AI."

## Notable framing choices

- The role is **cross-cutting**, not narrowly technical — it eats both eng and business-process work.
- It will be called **different things at different companies** — Levie deliberately doesn't name it.
- The growth is **outside tech** — non-tech enterprises will scale this fastest.

## Key takeaways

- "Bring AI to your team" is **not a side project**. It needs an owner.
- The job spec spans deploy + context + evals + HITL + upgrades + change management — far beyond prompt engineering.
- Non-tech enterprises will create a *new tier* of software-engineering-shaped jobs.
