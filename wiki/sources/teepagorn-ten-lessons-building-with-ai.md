# Ten Lessons from Building Bots, Toys, and Tools with AI / 10 บทเรียนจากการสร้างของด้วย AI

โพสต์จาก Facebook handle `@teepagorn` เล่าบทเรียนภาคสนามหลังผู้เขียนสร้าง bot ของเล่น และเครื่องมือแทบทุกวัน ใจความไม่ใช่ “AI เขียนโค้ดเก่งแค่ไหน” แต่เป็น **พอการลงมือเร็วขึ้น งานสำคัญของคนก็ย้ายไปอยู่ที่การออกแบบ การแบ่งงาน การให้บริบท และการตัดสินคุณภาพ**

ข้อมูลที่ผู้ใช้ส่งมาไม่มีชื่อผู้เขียน วันที่โพสต์ หรือ permalink ของโพสต์โดยตรง ส่วน title ที่ให้มาคือ “Search results” ซึ่งดูเป็นชื่อหน้า Facebook มากกว่าชื่อบทความ Wiki จึงใช้ชื่ออธิบายเนื้อหา และอ้างผู้เขียนเพียงด้วย handle โดยไม่เดาตัวบุคคล

## 1. Engineering และ system design ยังไม่หายไป

AI ลดภาระจำ syntax และพิมพ์ code ทีละบรรทัด แต่ไม่ได้รู้เองว่าระบบที่ดีในบริบทของเราควรหน้าตาอย่างไร คนยังต้องตัดสินว่าแบ่ง component แบบไหน ข้อมูลไหลทางไหน ส่วนใดคุยกัน และควรเก็บ state ตรงไหน

ระบบที่ AI สร้างอาจ “ทำงานได้” แต่ยังช้า ผิด หรือดูแลยาก เพราะความถูกต้องระดับ function ไม่เท่ากับความเหมาะสมระดับระบบ ประเด็นนี้ตรงกับ [[engineering-role-shift|บทบาทวิศวกรที่ย้ายไปต้นน้ำและปลายน้ำ]]

**ได้อะไร:** ความเร็วในการ generate code ยิ่งสูง งานออกแบบก่อนเขียนยิ่งมีผลต่อสิ่งที่ถูกสร้างตามมา

## 2. แตกงานให้เล็กพอจะสั่งและตรวจ

งานใหญ่ควรถูกซอยเป็นก้อนที่มีขอบเขตชัด แล้วสั่งและเช็กทีละก้อน ผู้เขียนเปรียบแต่ละก้อนกับตัวต่อ Lego การแบ่งอาจทำเป็นลำดับรวดเดียว หรือแบ่งเป็นหลาย phase ก็ได้

ถ้าสั่งทั้งก้อน AI อาจทำเสร็จเร็ว แต่คนต้องย้อนแกะว่าความผิดเริ่มตรงไหน การแตกงานจึงไม่ได้มีไว้ให้ AI เข้าใจอย่างเดียว มันทำให้จุดตรวจและจุดย้อนกลับเล็กลงด้วย

**ผลคือ:** งานใหญ่เปลี่ยนจากการเดิมพันครั้งเดียวเป็นวงรอบเล็กที่ตรวจได้

## 3. Folder structure คือแผนที่ร่วมของคนกับ AI

การตั้งชื่อไฟล์ แยก module และวางของไว้ใน folder ที่คาดเดาได้ ช่วยทั้งคนและ AI หา code เจอ ผู้เขียนเปรียบกับการจัดบ้านก่อนชวนคนอื่นมาช่วย ถ้าบ้านรก ต่อให้ผู้ช่วยเก่งก็เสียเวลาเดินหาและอาจสร้างของซ้ำ

นี่ทำให้ structure ไม่ใช่งานตกแต่งท้ายโครงการ แต่เป็นส่วนหนึ่งของ [[context-engineering|การจัดบริบท]] ตั้งแต่ต้น เพราะตำแหน่งและชื่อไฟล์ช่วยบอกความหมายโดยไม่ต้องอธิบายใหม่ทุก prompt

**ได้อะไร:** โครงสร้างที่อ่านออกลดทั้งเวลาค้นหาและโอกาสที่ AI จะเดาความสัมพันธ์ผิด

## 4. Context สำคัญพอ ๆ กับ prompt

AI ตอบได้ดีเท่ากับสิ่งที่มันเห็น ถ้าให้ไฟล์ กฎ และตัวอย่างที่เกี่ยวข้องครบ ผลมักดีขึ้น ถ้าให้ไม่พอ model ต้องเติมช่องว่างเอง และอาจเดาผิดด้วยน้ำเสียงมั่นใจ

ผู้เขียนตั้งข้อสังเกตอีกด้านว่า งานที่ต้องถ่ายทอด context สูงมากอาจไม่คุ้มจะ delegate เพราะเวลาอธิบายอาจนานกว่าทำเอง ตรงนี้เป็นข้อจำกัดของ [[context-engineering]] ไม่ใช่แค่ปัญหา prompt สั้นหรือยาว

**ผลคือ:** ก่อนสั่ง AI ต้องคิดทั้ง “จะพูดว่าอะไร” และ “มันต้องเห็นอะไรจึงทำงานนี้ได้”

## 5. Skill stacking ช่วยให้ต่อของข้ามโดเมน

ผู้เขียนเรียกข้อได้เปรียบนี้ว่า [[skill-stacking|skill stacking]] คือรู้หลายเรื่องในระดับพอทำได้ เช่น design, code, marketing และ psychology แล้วมองออกว่าจะเอาอะไรต่อกับอะไร จากนั้นใช้ AI เติมความลึกในแต่ละด้าน

ตัวเลข “70%” ในโพสต์เป็นภาพเปรียบเทียบ ไม่ใช่เกณฑ์วัดทักษะที่ผ่านการทดลอง และยังมีความตึงกับข้อ 7: ถ้าความรู้บางเกินไป คนอาจไม่รู้ว่า AI เติมรายละเอียดผิด

**ได้อะไร:** breadth ช่วยเลือกชิ้นส่วน แต่ depth ยังจำเป็นตรงจุดที่ต้องออกแบบและตรวจ

## 6. Taste มีค่าขึ้นเมื่อการสร้างถูกลง

เมื่อหลายคนสร้างของได้เร็วใกล้กัน สิ่งที่แยกของดีออกจากของกลาง ๆ คือ taste หรือสายตาที่ตัดสินว่าอะไรสวย ใช้ง่าย รก หรือควรตัดทิ้ง AI สร้าง variation ตามคำสั่งได้ แต่ไม่รู้เองว่าแบบไหน “ดี” สำหรับผู้ใช้และ product นี้

ผู้เขียนใช้ heuristic ว่า AI พางานไปได้ราว 70% ส่วนช่วง 70→90% ต้องอาศัยคนเกลา ตัด และลองซ้ำ ตัวเลขนี้เป็นประสบการณ์ส่วนตัว ไม่ใช่ benchmark แต่สอดคล้องกับ [[taste-paradox|Taste Paradox]] ที่ว่าคนต้องมี judgement ก่อนจึงจะคุม AI ได้ดี

**ผลคือ:** คอขวดใหม่ไม่ใช่การผลิต option แต่เป็นการเลือกและเกลา option ที่ควรอยู่ต่อ

## 7. ความรู้พื้นฐานคือเครื่องจับ hallucination

AI พูดตอนผิดได้มั่นใจเหมือนตอนถูก ถ้าคนไม่มี mental model ของงาน ก็อาจเชื่อ code หรือคำอธิบายที่ผิด และวันที่ระบบพังจะบอกจุดตรวจหรือซ่อมเองไม่ได้

ความรู้ยังช่วย direct การ debug ได้ เช่น ถ้าคนสงสัยว่า schema database ทำให้เกิด bug ก็ชี้ให้ AI ไปตรวจ path นั้นได้เร็วกว่าให้มันงมทั้งระบบ ตรงนี้เชื่อมกับ [[skill-atrophy|ความเสี่ยงที่ทักษะถดถอย]] เมื่อ delegate งานที่ตัวเองยังตรวจไม่ได้

**ได้อะไร:** ความรู้ของคนไม่ได้แข่งกับความจำของ model แต่ทำหน้าที่ตั้งสมมติฐานและไม่ยอมรับคำตอบที่ดูดีเกินจริง

## 8. เริ่มเล็ก สร้างให้ตัวเองใช้ และลองเร็ว

ผู้เขียนเสนอสามอย่างคู่กัน: ทำชิ้นเล็กที่สุดที่ใช้ได้ก่อน เป็นผู้ใช้คนแรกของตัวเอง และลดต้นทุนการทดลองให้ต่ำพอจะลองหลายแบบในหนึ่งชั่วโมง ของจริงทำให้ friction โผล่เร็วกว่าการวาดภาพทั้งหมดในหัว

เมื่อสร้างเร็ว การลบทางตันก็เจ็บน้อยลง สิ่งทดลองบางชิ้นยังเก็บกลับมาเป็น component ของระบบใหญ่ได้ แนวนี้ใกล้กับ [[throwaway-prototyping|prototype แบบทิ้งได้]] แต่ต่างนิดหนึ่งตรงที่บางชิ้นไม่ได้ทิ้งทั้งก้อน อาจกลายเป็นตัวต่อที่ reuse ได้หลังตรวจคุณภาพแล้ว

**ผลคือ:** ความเร็วมีค่าตอนใช้ซื้อ feedback ไม่ใช่ตอนใช้สะสม code ให้มากที่สุด

## 9. การรู้ว่าจะถามอะไรยากกว่าการหาคำตอบ

คำตอบถูกลงและหาเร็วขึ้น แต่การนิยามปัญหาจริงยังต้องใช้คน ผู้เขียนมองว่าทักษะใหม่คือรู้เส้นแบ่งว่างานไหนควรสั่ง AI และงานไหนอธิบายนานกว่าลงมือเอง

ตัวอย่าง failure ส่วนตัวคือการให้ AI แนะนำบทความจากอินเทอร์เน็ตที่ผู้เขียนน่าจะชอบในวันนั้น แม้จูนหลายรอบก็ยังไม่ถูกใจ ผู้เขียนตั้งสมมติฐานว่าอาจยังนิยามคำถามและ taste ที่ซ่อนอยู่ไม่ชัดพอ

**ได้อะไร:** bottleneck ขยับจาก answer retrieval ไปสู่ problem framing และการถ่ายทอด preference ที่พูดออกมายาก

## 10. AI เป็นตัวคูณของที่เรามี

ประโยคสรุปของโพสต์คือ “AI เหมือนเป็นตัวคูณสิ่งที่เรามีอยู่แล้ว” คนที่รู้เรื่องลงทุนจะถามลึก ตรวจคำตอบ และใส่ skill ด้านลงทุนให้ AI ได้ คนที่มีพื้นฐาน design ก็ผลักงานไปทาง design ได้ดีกว่า

ตัวอย่างปิดท้ายคือ นักบัญชีที่ใช้ AI สร้างโปรแกรมบัญชีน่าจะได้ระบบที่ถูกต้องและตอบโจทย์กว่าคนที่ไม่รู้บัญชีเลย ภาพนี้คล้าย claim ของ [[dora|DORA]] ว่า AI ขยาย magnitude ของสิ่งที่ระบบมีอยู่ แต่ไม่เลือก direction ให้เอง อย่างไรก็ดี โพสต์พูดระดับความสามารถของคน ส่วน DORA พูดระดับแนวปฏิบัติของทีม จึงยังไม่ควรถือเป็นหลักฐานชุดเดียวกัน

**ผลคือ:** AI ไม่ได้ทำให้ความสนใจและประสบการณ์เดิมหมดค่า แต่มักทำให้ leverage ของสิ่งเหล่านั้นสูงขึ้น

## ภาพรวมของ workflow ที่โพสต์เสนอ

เมื่อต่อทั้งสิบข้อเข้าด้วยกัน จะได้วงรอบประมาณนี้:

1. ใช้ความรู้โดเมนกับ taste เลือกปัญหาที่ควรทำ
2. ออกแบบระบบและจัด structure ให้มองทางออก
3. แตกงานเป็นก้อนเล็ก พร้อม context และจุดตรวจ
4. ให้ AI สร้าง option หรือ implementation อย่างรวดเร็ว
5. ใช้ของจริงเอง ตรวจ hallucination และเกลาจาก 70% ไปข้างหน้า
6. ลบทางตัน เก็บชิ้นที่ดี แล้ววนรอบใหม่

นี่เป็น [[agentic-engineering|Agentic Engineering]] แบบคนเดียวที่เน้น judgement มากกว่า agent swarm หรือ orchestration ซับซ้อน

## ความตึงและคำถามที่ยังเปิดอยู่

- **Breadth vs depth:** [[skill-stacking]] แบบรู้หลายด้านราว 70% พอให้ direct และตรวจ AI จริงหรือไม่ หรือจำเป็นต้องมี anchor domain ที่ลึกอย่างน้อยหนึ่งด้านตาม [[taste-paradox]]
- **AI เติมของที่ไม่มีได้แค่ไหน:** ข้อ 5 บอกว่า AI เติมความลึกใน skill ที่เรารู้พอเปิด map ได้ แต่ข้อ 10 บอกว่า AI ไม่ได้เติมของที่เราไม่มีตั้งแต่ต้น เส้นแบ่งระหว่าง “ช่วยค้นเพื่อเรียน” กับ “สร้างแทนความเข้าใจ” อยู่ตรงไหน
- **Context แพงเมื่อไร:** งาน high-context ควรทำเองเสมอ หรือควรลงทุนทำ docs, tests และ harness เมื่อ context นั้นจะถูกใช้ซ้ำหลายครั้ง
- **70→90% วัดอย่างไร:** งานแต่ละชนิดมี verifier ต่างกัน ตัวเลขจากประสบการณ์ส่วนตัวจึงยังบอกไม่ได้ว่า quality gap เหลือเท่าไรจริง
- **สร้างให้ตัวเองก่อนมีข้อจำกัดอะไร:** self-use ช่วยจับ friction ได้ดี แต่ product สำหรับคนกลุ่มอื่นยังต้องมี research และ feedback จากผู้ใช้จริง

## ข้อจำกัดของแหล่ง

- เป็นบันทึกประสบการณ์ของคนคนเดียว ไม่ใช่งานทดลองหรือ benchmark
- ข้อความที่ให้มาไม่มีวันที่โพสต์ ชื่อผู้เขียน หรือ permalink ระดับโพสต์
- ตัวเลข 70% เป็น heuristic เชิงประสบการณ์ ไม่มี rubric หรือชุดงานรองรับ
- ตัวอย่างส่วนใหญ่เป็นงานสร้าง bot ของเล่น และเครื่องมือ จึงอาจไม่ครอบคลุมระบบ regulated, safety-critical หรือทีมขนาดใหญ่
