# Large-Scale Changes (LSC) / การเปลี่ยนแปลงขนาดใหญ่

ความสามารถทางวิศวกรรมขั้นสูง (หรือแนวทางปฎิบัติ) ที่อนุญาตให้นักพัฒนาซอฟต์แวร์เพียงคนเดียว สามารถทำการแก้ไขหรือปรับปรุงโครงสร้างโค้ดแบบครอบคลุม (coordinated changes) ในระดับหลายล้านบรรทัด ข้ามไฟล์ซอฟต์แวร์นับพันแอปพลิเคชันทั่วทั้งองค์กรได้อย่างพร้อมเพรียงและปลอดภัย

แนวคิดนี้ได้รับการพัฒนาขึ้นและใช้งานเป็นปกติภายใน Google มายาวนานกว่า 15 ปี (อ้างอิงจากการบรรยายของ [[adam-bender|Adam Bender]] ในปี 2026) โดย LSC ถือเป็นเสาหลักทางเทคโนโลยีที่ช่วยให้โค้ดเบสขนาดมหึมาของ Google สามารถคงความสดใหม่ ทันสมัย และสามารถอัปเกรดเวอร์ชันของภาษาหรือเฟรมเวิร์กไปพร้อมกันได้โดยไม่ติดหล่มหนี้สินทางเทคโนโลยี (Technical Debt)

---

## แก่นความคิด

LSC ไม่สามารถเกิดขึ้นได้จากการมีแค่เครื่องมือเขียนโค้ดอัตโนมัติ แต่มันเรียกร้ององค์ประกอบและรากฐานของ [[software-ecology|ระบบนิเวศซอฟต์แวร์]] ทั้งหมดทำงานประสานกัน:

1.  **รากฐานทางโครงสร้าง (Shared Fate):** โค้ดทั้งหมดต้องรวมอยู่ใน **Monorepo** เพื่อให้ง่ายต่อการค้นหาและเข้าถึงโค้ดส่วนที่ต้องการแก้ไขพร้อมกันในคราวเดียว
2.  **ระบบทดสอบอัตโนมัติระดับโลก (Global Test Automation Platform):** เมื่อมีการแก้ไขโค้ดนับล้านบรรทัด สมองมนุษย์ไม่มีทางเข้ามาตรวจสอบความเสียหายได้ ระบบวิศวกรรมจึงต้องการเครื่องมือรันเทสระดับอัตโนมัติที่มีประสิทธิภาพสูง รันเทสระดับพันล้านครั้งต่อวัน เพื่อยืนยันว่าโค้ดที่ถูกแก้นั้นไม่ทำลายแอปพลิเคชันปลายทางของคนอื่น
3.  **ความสม่ำเสมอของเครื่องมือ (Universal Build Toolchain):** ทุกทีมในองค์กรต้องใช้เครื่องมือคอมไพล์และบิลด์ซอฟต์แวร์เดียวกัน เพื่อให้มั่นใจได้ว่าโค้ดที่รันผ่านในขั้นตอน LSC จะทำงานได้ผลลัพธ์เหมือนกันทุกประการในเครื่องของทีมอื่นๆ
4.  **วัฒนธรรมการลดงานซ้ำซาก (Automation over Toil):** มุ่งเน้นการสร้างสคริปต์อัตโนมัติมาแก้โค้ดแทนการกะเกณฑ์ให้มนุษย์เข้าไปนั่งกดแก้ทีละไฟล์ ซึ่งถือเป็น "งานถึกเหนื่อยยากอันเปล่าประโยชน์" (Toil)

---

## ความท้าทายในยุยุค AI

ในอดีต ความสามารถในการทำ LSC จำกัดอยู่เฉพาะในบริษัทเทคโนโลยียักษ์ใหญ่อย่าง Google ที่มีโครงสร้างพื้นฐานพร้อม แต่เมื่อก้าวเข้าสู่ **[[10x-moment|ช่วงเวลา 10 เท่า (10x Moment)]]** ของ AI เครื่องจักรสร้างโค้ดทำให้นักพัฒนาทุกคนในทุกบริษัทสามารถสั่งการให้ AI เขียนสคริปต์ปรับปรุงโค้ดเบสขนาดใหญ่ได้ทันที ความท้าทายใหม่ที่อุบัติขึ้นจึงรวมถึง:

-   **สงครามความขัดแย้งในการผสานโค้ด (Super-Large Merge Conflicts):** เมื่อวิศวกรทุกคน (และ Agent ทุกตัว) สามารถพ่น LSC ออกมาแก้ไฟล์ทีละหลายหมื่นบรรทัดพร้อมๆ กัน โค้ดเหล่านั้นจะวิ่งชนกันจนเกิด Conflict มหาศาลระดับนับหมื่นบรรทัดที่เครื่องมือคุมเวอร์ชันแบบเดิมๆ รับมือไม่ได้
-   **สงครามแก้ไขโค้ดระหว่าง Agent (Agentic Edit Wars):** อาจเกิดสถานการณ์ที่ Agent ตัวแรกทำ LSC เพื่อปรับสไตล์โค้ดแบบหนึ่ง แต่มี Agent อีกค่ายเข้ามาสั่ง LSC ทับเพื่อแก้อีกแบบ ทำให้บริษัทต้องสูญเสียค่าโทเคนประมวลผลและการรันเทสไปกับศึกของเครื่องจักรอย่างไร้ค่า
-   **ความจำเป็นของข้อตกลงทางสังคม (Social Contracts):** องค์กรจำเป็นต้องพัฒนาเครื่องมือจัดลำดับความสำคัญและข้อตกลงประชาคมของวิศวกรและ Agent เพื่อให้แต่ละฝ่ายทำงานหลบหลีกและสอดคล้องกัน

**Why this helps:** การออกแบบระบบให้รองรับ LSC อย่างมีระบบและมีความปลอดภัย จะเป็นปราการสำคัญที่ช่วยให้องค์กรสามารถควบคุมคุณภาพและสลายโครงสร้างโค้ดเก่าในยุคที่ความเร็วของ AI พุ่งทะยานได้ โดยไม่ตกหลุมพรางของการสเกลอย่างไร้ทิศทาง

## LSC ในมือ agent ตัวเดียว: dynamic workflows

ตัวอย่างที่ Anthropic ยกตอนเปิด [[claude-opus-4-8|Opus 4.8]] คือ [[claude-code|Claude Code]] กับ [[dynamic-workflows|dynamic workflows]] **migrate codebase หลายแสนบรรทัดได้ตั้งแต่เริ่มจนถึง merge** โดยใช้ test suite ที่มีอยู่เป็นเกณฑ์ตัดสิน — เป็นการดึงความสามารถ LSC ที่เคยต้องอาศัยทีม + โครงสร้างพื้นฐานแบบ Google เข้ามาไว้ใน agent ตัวเดียวที่กระจายเป็น subagent คู่ขนานเป็นร้อย หัวใจยังเหมือนเดิม: **automated test เป็น oracle** ตัดสินว่าการเปลี่ยนแปลงพังของคนอื่นไหม
