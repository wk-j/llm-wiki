# Model Choice by Domain Expertise / เลือก model ให้พอดีกับคนใช้

[[somkiat-khitwongwattana|Akexorcist]] ได้โพสต์บน Facebook (2026-04-21) ว่าการเลือก model ไม่จำเป็นต้องดูว่าตัวไหนใหม่ที่สุด แต่ควรพิจารณาจากผู้ใช้ก่อน หากผู้ใช้มีความสามารถพอที่จะตรวจสอบโค้ดเองได้, model รุ่นเก่าก็อาจเพียงพอ แต่หากผู้ใช้ต้องการ "vibe" (ทำงานแบบไม่ตรวจสอบ), ควรจ่ายเพิ่มเพื่อใช้ model รุ่นใหม่มาช่วยลดความผิดพลาด

ในหน้านี้, คำว่า **taste** หมายถึง *วิจารณญาณหรือสายตาที่สามารถบอกได้ว่าโค้ดใดใช้ได้หรือไม่ได้* — ซึ่งเป็นสิ่งที่ได้มาจากการลงมือทำจริงจนเข้าใจอย่างลึกซึ้ง (ดู [[taste-paradox]] ของ [[nattee-niparnan|Nattee]] ที่ขยายความไว้)

## แก่นของแนวคิด

คำถามที่ว่า "ใช้ model ไหนดี" ยังไม่สมบูรณ์ ต้องถามต่อว่า "คนใช้เก่งแค่ไหน" ด้วย เพราะคุณภาพของงานขึ้นอยู่กับทั้งสองปัจจัยรวมกัน ไม่ใช่แค่ตัว model เพียงอย่างเดียว

Somkiat ได้แบ่งออกเป็นสองกรณี:

| สถานการณ์ | คอขวดของคุณภาพ | model ที่เหมาะสม |
|---|---|---|
| ผู้ใช้ลงไปอ่านโค้ดเอง + เป็น Domain Expert | อยู่ที่คน — แต่คนมี "ต่อมเอ๊ะ" อยู่แล้ว | [[claude\|Claude Sonnet 4.5]] เพียงพอสำหรับงานส่วนใหญ่ |
| Vibe — ไม่เปิดโค้ดดู / ไม่มีพื้นฐานในเรื่องนั้น | อยู่ที่ model — เพราะไม่มี "ต่อมเอ๊ะ" ของคนมาช่วย | ใช้ model ที่ใหม่ที่สุดเท่าที่จ่ายไหว |

> "ใช้แค่ Claude 4.5 Sonnet ก็เพียงพอกับงานส่วนใหญ่แล้วนะ มีน้อยมากที่ต้องใช้ Model ที่ใหม่กว่านี้ ... แต่ถ้าเป็นงานที่ต้องการ Vibe ... การเลือกใช้ Model ใหม่ ๆ ไว้ก่อน อันนี้ก็เข้าใจได้" — Somkiat

**ข้อสรุป:** แนวคิดที่ว่า "รุ่นใหม่ต้องดีกว่าเสมอ" ต้องถูกปรับเป็น "รุ่นใหม่ช่วยได้มากที่สุดเมื่อผู้ใช้ไม่มีความรู้ในเรื่องนั้น" คนที่มี taste อยู่แล้วสามารถประหยัดเงินและเวลาได้โดยที่คุณภาพงานไม่ลดลง

## ทำไมถึงเป็นเช่นนี้

การจะได้งานที่ดีต้องมีสององค์ประกอบ: generation (การสร้างโค้ดที่ถูกต้อง) และ judgment (การรู้ว่าโค้ดที่สร้างขึ้นมานั้นใช้ไม่ได้)

เมื่อ model ออกรุ่นใหม่, ทั้งสองด้านนี้จะเก่งขึ้น แต่ไม่เท่ากัน:

-   **ฝั่ง generation** — model รุ่นปี 2025–2026 มีความสามารถเพียงพอสำหรับงานพัฒนาทั่วไปอยู่แล้ว และรุ่นใหม่กว่าก็ไม่ได้ดีขึ้นมากนัก
-   **ฝั่ง judgment** — ยังไม่เก่งพอที่จะปล่อยให้ตัดสินใจเองได้ และยังต้องมีคนคอยตรวจสอบอยู่

เมื่อเปรียบเทียบสองกรณี:

-   **คนตรวจมี taste (Domain Expert)** — judgment ของคนเต็มเปี่ยมอยู่แล้ว ไม่จำเป็นต้องให้ model มาตรวจตัวเองซ้ำ การใช้ Sonnet 4.5 ก็เพียงพอ
-   **คนไม่ตรวจ / ไม่มี taste (Vibe)** — judgment ของคนเป็นศูนย์ ทำให้เหลือแต่ model ที่ต้องตัดสินใจเอง จึงต้องใช้ model ที่สามารถตรวจสอบตัวเองได้ดี เช่น [[claude-opus-4-7|Opus 4.7]] ที่ Anthropic อ้างว่ามีการตรวจสอบตัวเองที่ดีขึ้น เพื่อลดโอกาสเกิด bug

**ข้อสรุป:** model ใหม่ที่ราคาสูง ไม่ได้แพงแค่ค่า token — แต่สิ่งที่จ่ายเพิ่มคือค่าเช่า "ต่อมเอ๊ะ" ของ model คนที่มี "ต่อมเอ๊ะ" ของตัวเองอยู่แล้วจึงไม่จำเป็นต้องจ่ายค่าเช่านี้

## ความเชื่อมโยงกับ [[taste-paradox]]

หลักการของ Somkiat และ taste paradox ของ [[nattee-niparnan|Nattee]] พูดถึงเรื่อง taste เหมือนกัน แต่มองคนละด้าน:

| | Somkiat (การเลือก model) | Nattee ([[taste-paradox]]) |
|---|---|---|
| **คำถาม** | เมื่อมี taste แล้ว ควรใช้ model ไหน | taste เกิดขึ้นได้อย่างไร |
| **มุมมอง** | การใช้งานจริง — ใช้ taste ที่มีอยู่ให้คุ้มค่า | หลักสูตร / ระยะยาว — taste จะมาจากไหน หาก Agent ทำให้ข้ามการฝึกฝน |
| **คำตอบ** | คนมี taste → รุ่นเก่าพอ; vibe → จ่ายเพิ่ม | ยังไม่มีข้อสรุป — *"ต้องฝึกฝนลึกแค่ไหนถึงจะพอ?"* |

เมื่อนำมาต่อกันจะเห็นภาพที่สมบูรณ์: Somkiat ตอบว่า "ถ้ามี taste แล้วจะทำอย่างไรให้คุ้มค่า" ในขณะที่ Nattee ถามว่า "จะเอา taste มาจากไหนตั้งแต่แรก" หากฝั่งของ Nattee ยังไม่มีคำตอบ, เด็กรุ่นใหม่อาจถูกผลักดันให้กลายเป็น vibe coder และต้องจ่ายค่า "เช่า taste" ให้กับ model ใหม่ไปเรื่อยๆ

## ยังมีแกนอื่นในการปรับแต่ง AI

การเปลี่ยน model เป็นเพียงแกนหนึ่งในการปรับแต่ง AI ให้เหมาะสมกับงาน ยังมีแกนอื่นๆ อีก:

-   **[[effort-levels]]** — ปรับ effort (low/high/xhigh/max) บน model เดิม โดยไม่ต้องเปลี่ยน tier
-   **[[task-budgets]]** — ให้ model เดิมคิดนานขึ้นหรือเร็วขึ้นผ่าน task-budget
-   **[[delegation-mindset]]** — หากเป็น Domain Expert, ควรใส่ intent และ acceptance criteria ให้ครบตั้งแต่ turn แรก
-   **[[auto-mode]]** — หากสามารถตรวจสอบทุก tool call ได้ ให้ใช้ default แต่หากเน้น vibe, ให้เปิด auto-mode เพื่อให้ classifier กรองก่อน

**ข้อสรุป:** เมื่อเจอปัญหา, คนที่ไม่คุ้นเคยอาจรีบสรุปว่า "ต้องอัปเกรด model" แต่จริงๆ แล้วแกนอื่นอาจถูกกว่าและเร็วกว่ามาก — เช่น การเพิ่ม effort, การให้ context ที่ดี, หรือการใช้ advisor/subagent มาช่วยตรวจสอบ

## ข้อควรระวัง

-   **ใช้ได้กับงานที่ model รุ่นก่อนสามารถทำได้อยู่แล้ว** — สำหรับงานที่ต้องการ capability ที่สูงกว่า (เช่น การดูรูป 2000+ px ที่ [[claude-opus-4-7]] เพิ่งเปิดตัว) ยังคงต้องใช้รุ่นใหม่
-   **"Domain Expert" ไม่ใช่ 0/1** — senior ในโดเมนหนึ่งอาจเป็น vibe coder ในอีกโดเมนได้ ควรพิจารณาเป็นรายกรณี
-   **มีต้นทุนแฝง** — การประหยัดค่า token จาก Sonnet 4.5 อาจต้องแลกมาด้วยเวลาของ senior ที่ต้องมาตรวจสอบ ซึ่งต้องชั่งน้ำหนักให้ดี (ดู [[judgement-vs-automation]])
-   **Self-verification ของ model ใหม่ไม่ใช่เวทมนตร์** — [[claude-opus-4-7]] ก็ยังสามารถพลาดได้ การใช้ vibe + Opus ไม่ได้หมายความว่าจะปลอดภัย 100% แต่แค่ปลอดภัยกว่า vibe + Sonnet
