# Model Choice by Domain Expertise / เลือก model ให้พอดีกับคนใช้

[[somkiat-khitwongwattana|Akexorcist]] โพสต์ไว้บน Facebook (2026-04-21) ว่าเลือก model ไม่ต้องดูว่าตัวไหนใหม่สุด ให้ดูคนใช้ก่อน ถ้าคนเก่งพอจะนั่งตรวจโค้ดเองได้ รุ่นเก่าก็พอแล้ว แต่ถ้าคนจะ vibe ไม่ตรวจเอง ค่อยจ่ายเพิ่มให้รุ่นใหม่มาช่วยอุดแทน

ทั้งหน้านี้คำว่า **taste** หมายถึง *ต่อมเอ๊ะ / สายตาที่บอกได้ว่าโค้ดอันไหนใช้ได้ อันไหนใช้ไม่ได้* — เป็นสิ่งที่ได้มาจากการเคยลงมือทำจนรู้ถึงกระดูก (ดู [[taste-paradox]] ของ [[nattee-niparnan|Nattee]] ที่ขยายความไว้)

## แก่น

ถามว่า "ใช้ model ไหนดี" ยังตอบไม่ครบ ต้องถามต่อว่า "คนใช้เก่งแค่ไหน" ด้วย เพราะงานจะออกมาดีหรือไม่ดี ขึ้นกับทั้งสองอย่างรวมกัน ไม่ใช่แค่ตัว model

Somkiat แยกให้ดูสองกรณี:

| สถานการณ์ | คอขวดของคุณภาพอยู่ตรงไหน | model ที่พอ |
|---|---|---|
| ลงไปอ่านโค้ดเอง + เป็น Domain Expert | อยู่ที่คน — แต่คนมีต่อมเอ๊ะอยู่แล้ว | [[claude\|Claude Sonnet 4.5]] พอสำหรับงานส่วนใหญ่ |
| Vibe — ไม่เปิดโค้ดดู / ไม่มีพื้นในเรื่องนั้น | อยู่ที่ model — เพราะไม่มีต่อมเอ๊ะของคนมาช่วย | เอาตัวที่ใหม่สุดเท่าที่จ่ายไหว |

> "ใช้แค่ Claude 4.5 Sonnet ก็เพียงพอกับงานส่วนใหญ่แล้วนะ มีน้อยมากที่ต้องใช้ Model ที่ใหม่กว่านี้ ... แต่ถ้าเป็นงานที่ต้องการ Vibe ... การเลือกใช้ Model ใหม่ ๆ ไว้ก่อน อันนี้ก็เข้าใจได้" — Somkiat

**ได้อะไร:** วิธีคิดปกติของวงการที่ว่า "รุ่นใหม่ต้องดีกว่าเสมอ" ต้องแก้ใหม่ว่า "รุ่นใหม่ช่วยได้แค่ตอนที่คนไม่รู้เรื่อง" คนที่มี taste เองก็ประหยัดเงินประหยัดเวลาไปเลย งานก็ไม่ได้แย่ลง

## ทำไมถึงเป็นแบบนี้

จะให้งานออกมาดี ต้องมีของสองอย่างพร้อมกัน — อย่างแรกคือ generation (พ่นโค้ดให้ถูก) อย่างที่สองคือ judgment (รู้ว่าที่พ่นออกมาอันไหนใช้ไม่ได้)

พอ model ออกรุ่นใหม่ ทั้งสองฝั่งก็เก่งขึ้นจริง แต่ขึ้นไม่เท่ากัน:

- **ฝั่ง generation** — model รุ่นปี 2025–2026 เก่งถึงจุดที่ทำงาน dev ทั่วไปได้โอเคอยู่แล้ว รุ่นใหม่กว่าก็ดีขึ้นไม่เท่าไหร่
- **ฝั่ง judgment** — ยังไม่เก่งพอให้ปล่อยตัดสินเอง คนก็ต้องคอยตรวจอยู่ดี

ทีนี้ลองเทียบสองกรณี:

- **คนตรวจมี taste เอง (Domain Expert)** — judgment ฝั่งคนเต็มอยู่แล้ว ไม่ต้องง้อให้ model มาตรวจตัวเองซ้ำ ฝั่ง generation เอา Sonnet 4.5 ก็พอ
- **คนไม่ตรวจ / ไม่มี taste (Vibe)** — judgment ฝั่งคนเป็นศูนย์ เหลือแต่ model ตัดสินเอง เลยต้องหาตัวที่ตรวจตัวเองเก่ง ๆ เช่น [[claude-opus-4-7|Opus 4.7]] ที่ Anthropic ชูว่าตรวจตัวเองดีขึ้น จะได้ลดโอกาส bug หลุด

**ได้อะไร:** จะมองเห็นว่า model ใหม่ที่ว่าแพง ไม่ได้แพงแค่ค่า token — ที่จ่ายเพิ่มจริง ๆ คือค่าเช่า "ต่อมเอ๊ะ" ของ model คนที่มีต่อมเอ๊ะของตัวเองอยู่แล้ว ไม่ต้องเสียตังเช่า

## เชื่อมกับ [[taste-paradox]] — ถามคนละด้านของเรื่องเดียวกัน

หลักของ Somkiat กับ taste paradox ของ [[nattee-niparnan|Nattee]] พูดเรื่อง taste เหมือนกัน แต่มองคนละด้าน:

| | Somkiat (เลือก model) | Nattee ([[taste-paradox]]) |
|---|---|---|
| ถาม | พอมี taste แล้ว ใช้ model ไหนดี | แล้ว taste มันเกิดขึ้นได้ยังไง |
| มุม | เวลาใช้งานจริง — ใช้ taste ที่มีอยู่ให้คุ้ม | หลักสูตร / ระยะยาว — taste จะมาจากไหน ถ้า Agent ให้ข้ามการฝึกมือ |
| ตอบ | คนมี taste → รุ่นเก่าพอ; vibe → จ่ายเพิ่ม | ยังคิดไม่ตก — *"ต้องฝึกมือลึกแค่ไหนถึงพอ?"* |

เอามาต่อกันก็เป็นเส้นเดียว Somkiat ตอบว่า "ถ้ามี taste แล้วทำยังไงให้คุ้ม" ส่วน Nattee ถามว่า "จะเอา taste มาจากไหนตั้งแต่แรก" ถ้าฝั่ง Nattee ยังตอบไม่ได้ เด็กรุ่นใหม่ก็จะโดนดันให้กลายเป็น vibe coder ไปเลย แล้วก็ต้องจ่ายค่า "เช่า taste" ให้ model ใหม่ไปเรื่อย ๆ

## ยังมีแกนอื่นให้จูน AI นอกจากเปลี่ยน model

เปลี่ยน model เป็นแค่แกนเดียวของการจูน AI ให้พอดีงาน จริง ๆ ยังมีแกนอื่นอีก:

- **[[effort-levels]]** — ปรับ effort (low/high/xhigh/max) บน model เดิม ไม่ต้องขึ้น tier
- **[[task-budgets]]** — ให้ model เดิมคิดนานขึ้นหรือเร็วขึ้นผ่าน task-budget
- **[[delegation-mindset]]** — ถ้าเป็น Domain Expert ก็ใส่ intent กับ acceptance criteria ให้ครบตั้งแต่ turn แรก ไม่ต้องปล่อยให้ model เดาแล้วค่อยตามแก้
- **[[auto-mode]]** — ถ้าไหวจะนั่งตรวจทุก tool call ก็ใช้ default ถ้าเน้น vibe ค่อยเปิด auto-mode ให้ classifier คอยกรองก่อน

**ได้อะไร:** พอเจอปัญหาทีไร คนที่ยังไม่ชินมักรีบตอบว่า "ต้องอัปรุ่น model" จริง ๆ แกนอื่นถูกกว่าและเร็วกว่าเยอะ — ดัน effort ขึ้น ใส่ context ให้ดี หรือให้ advisor/subagent มาช่วยตรวจ

## ข้อควรระวัง

- **ใช้ได้กับงานที่ model รุ่นก่อนทำไหวอยู่แล้ว** — งานที่ capability ยังไม่พอจริง ๆ (เช่นดูรูป 2000+ px ที่ [[claude-opus-4-7]] เพิ่งเปิด) ยังไงก็ต้องไปรุ่นใหม่
- **"Domain Expert" ไม่ใช่ 0/1** — senior ในโดเมนหนึ่งก็เป็น vibe coder ในอีกโดเมนได้ แกนนี้ต้องมองเป็นรายงาน ไม่ใช่รายคน
- **มีต้นทุนแฝงที่มองไม่เห็น** — ประหยัดค่า token Sonnet 4.5 ได้ แต่อาจแลกกับเวลาของ senior ที่ต้องมานั่งตรวจ ต้องชั่งน้ำหนักดี ๆ (ดู [[judgement-vs-automation]])
- **Self-verification ของ model ใหม่ไม่ใช่เวทมนตร์** — [[claude-opus-4-7]] ก็ยังพลาดได้ vibe + Opus ไม่ได้แปลว่าปลอดภัย แค่ปลอดภัยกว่า vibe + Sonnet เฉย ๆ
