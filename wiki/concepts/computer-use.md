# Computer Use / การให้ Agent ใช้คอมพิวเตอร์

Computer use คือความสามารถที่ให้ AI agent มองหน้าจอ ขยับเมาส์ คลิก พิมพ์ และทำงานผ่านแอปเหมือนคน แทนที่จะเรียก API โดยตรงอย่างเดียว ใน [[gpt-5-6-and-openai-build-week-aimeowyak|ไลฟ์รีวิว GPT-5.6]] ความสามารถนี้ถูกวางเป็นจุดเด่นของ [[openai-codex|Codex]] เพราะทำงานเบื้องหลังและเปิดหลายงานคู่ขนานได้

## ทำไม UI automation ยังสำคัญ

หลายระบบไม่มี API หรือมีข้อมูลอยู่หลัง session ที่ผู้ใช้ล็อกอินไว้ เช่น เว็บข่าวแบบ subscription, ระบบมหาวิทยาลัย หรือหน้าดาวน์โหลด paper ตัว agent จึงใช้สิทธิ์เดียวกับผู้ใช้แล้วกดผ่าน UI จริง

ตัวอย่างในไลฟ์คือ:

- เปิดเว็บข่าวที่ผู้ใช้สมัครไว้ แล้วสรุปข่าวเป็น PowerPoint
- เปิดหลายเว็บไซต์พร้อมกันเพื่อเก็บข้อมูล
- ใช้สิทธิ์สถาบันของผู้ใช้ดาวน์โหลด paper ฉบับเต็มหลายรายการ
- ทดสอบ web app ด้วยการคลิกจริง

**ได้อะไร:** งานที่เคยติดตรง “ไม่มี API” ยัง automate ได้ ถ้าคนมีสิทธิ์เข้าถึงและยอมให้ agent ใช้ session นั้น

## คอขวดเปลี่ยนจาก model ไปอยู่ที่ environment

พอ model คลิกได้เร็วขึ้น ข้อจำกัดกลับย้ายไปที่เครื่อง, browser, network, captcha, rate limit และหน้าจอที่เปลี่ยนระหว่างรัน บน Mac ผู้พูดรายงานว่า Codex ทำงานเบื้องหลังได้เร็วกว่าเครื่อง Windows ที่เขาลอง แต่ยังเป็นประสบการณ์เฉพาะราย

**ผลคือ:** คุณภาพ computer use ไม่ได้มาจาก model อย่างเดียว ต้องดู harness, OS, session state และความเสถียรของ UI ด้วย

## สิทธิ์เข้าถึงไม่เท่ากับสิทธิ์ทำทุกอย่าง

จุดเสี่ยงคือ agent เห็นหน้าจอเดียวกับผู้ใช้และอาจกด action ที่มีเงินจริง ข้อมูลลับ หรือผลกระทบภายนอก ยิ่ง model มี agency สูงและพยายามปิดงานเอง ก็ยิ่งต้องกำหนดขอบเขตชัด

วิธีคุมที่ควรมี:

- จำกัดเว็บไซต์ โฟลเดอร์ และ account ที่เข้าได้
- ตั้งเพดานค่าใช้จ่ายและจำนวน action
- ขอคนยืนยันก่อนซื้อ ส่ง ลบ หรือเผยแพร่
- เก็บ log/screenshot เพื่อย้อนดูว่า agent ทำอะไร
- แยกงานอ่านข้อมูลออกจากงานที่แก้สถานะภายนอก

**ได้อะไร:** ให้ agent ทำงานคลิกซ้ำๆ ได้เต็มที่ แต่เก็บ action แพงและย้อนยากไว้หลัง human approval

## คำถามที่ยังเปิดอยู่

- ความเร็วและความแม่นต่างกันแค่ไหนระหว่าง OS และ browser
- เมื่อเปิดหลาย agent พร้อมกัน RAM/token จะโตอย่างไร
- จะพิสูจน์ได้อย่างไรว่า agent ใช้เฉพาะสิทธิ์ที่ผู้ใช้ตั้งใจให้ใช้
- งานแบบไหนควรใช้ UI และงานแบบไหนควรกลับไปใช้ API ที่ตรวจสอบง่ายกว่า
