# Conway's Law / กฎของคอนเวย์

ทฤษฎีการจัดการเชิงโครงสร้างทางสังคมและวิศวกรรม ที่สังเกตพบว่า **"สถาปัตยกรรมและผลงานออกแบบเทคโนโลยีขององค์กรใดๆ จะลอกแบบและสะท้อนหน้าตาโครงสร้างการสื่อสารภายในองค์กรนั้นๆ"** เสนอขึ้นครั้งแรกโดย Melvin Conway (โปรแกรมเมอร์และสถาปนิกซอฟต์แวร์) ในปี 1967

กฎนี้ถือเป็นสัจธรรมสำคัญในแวดวงวิศวกรรมซอฟต์แวร์ที่เตือนสติว่า การสร้างระบบเทคนิคัลไม่เคยแยกขาดจากกระบวนการและพฤติกรรมทางสังคมของคนทำงาน การพยายามฝืนกฎนี้โดยไม่ปรับโครงสร้างคน จะนำไปสู่ความล้มเหลวในการจัดระเบียบซอฟต์แวร์เสมอ

---

## แก่นความคิด

Melvin Conway ระบุหลักการไว้อย่างกระชับว่า:

> "Organizations which design systems are constrained to produce designs which are copies of the communication structures of these organizations."

ยกตัวอย่างให้เห็นภาพง่ายๆ ในงานเขียนโค้ด:

-   **ตัวอย่างเชิงคลาสสิก:** หากองค์กรจัดตั้งทีมงานย่อย 4 ทีมเพื่อร่วมกันสร้างคอมไพเลอร์ (ตัวแปลภาษา) ระบบสุดท้ายที่ได้มักจะเป็นคอมไพเลอร์ที่ประมวลผล 4 ขั้นตอน (four-pass compiler) เสมอเพื่อให้แต่ละทีมมีขอบเขตรอบการส่งต่อข้อมูลสอดคล้องกับขอบเขตงานของตัวเอง
-   **สถาปัตยกรรมตามผังองค์กร:** หากทีมงานแบ่งแยกเป็นทีม Frontend และ Backend ข้อมูลและฟังก์ชันการทำงานในโค้ดจะถูกหั่นแบ่งแยกออกจากกันอย่างรุนแรงและเกิดการเชื่อมโยงส่งค่าผ่าน API ในรูปแบบที่สะท้อนช่องทางคุยงานระหว่างสองหัวหน้าทีมนั้นๆ

## ความสำคัญในยุค AI และการปรับผังองค์กรใหม่

ในกระแสธารการเปลี่ยนแปลงของเครื่องมือและพฤติกรรมการพัฒนาซอฟต์แวร์ยุค AI กฎของคอนเวย์กลับมาทวีความสำคัญยิ่งขึ้นในมุมมองของ **[[socio-technical-system|ระบบสังคม-เทคนิค]]**:

1.  **การจัดองค์กรใหม่รอบโมเดล (Reorganize Around Models):** ดังแนวคิดที่ [[thorsten-ball|Thorsten Ball]] เสนอใน **[[software-after-software|Software After Software]]** ว่าองค์กรที่ต้องการเอาชนะความเฉื่อยชา ต้องยุบทีมขนาดใหญ่แบบเดิมแล้วหันมาตั้งทีมขนาดเล็กที่พึ่งพา AI หลายสิบตัวแทน (Frontier Camp) การปรับโครงสร้างองค์กรให้กะทัดรัดและใช้ Agent เป็นหน่วยสื่อสารนี้ จะสะท้อนไปสู่สถาปัตยกรรมซอฟต์แวร์ยุคใหม่ที่เป็นแบบ Decoupled (แยกชิ้นส่วนอย่างอิสระสูง) และจัดวางรูปแบบ API คล้ายคลึงกับช่องทางการคุยกันของ Agent
2.  **Inverse Conway Maneuver (กลยุทธ์ย้อนคอนเวย์):** คือระเบียบวิธีปฎิบัติที่นักวิจัยและสถาปนิกนิยมนำมาใช้ โดยแทนที่จะปล่อยให้โครงสร้างทีมดั้งเดิมเข้ามากำหนดหน้าตาโค้ดเบสที่เสื่อมโทรม เราจะทำการ "ออกแบบโครงสร้างทีมงานที่เราต้องการใหม่ก่อน" (เช่น ออกแบบให้เกิดทีมที่ดูแลเซอร์วิสขนาดสั้นกะทัดรัดแบบแยกชิ้น) เพื่อบีบบังคับทางสังคมให้สถาปัตยกรรมซอฟต์แวร์เปลี่ยนโฉมหน้าไปในทิศทางนั้นโดยธรรมชาติ

**Why this helps:** กฎของคอนเวย์ชี้ให้เราตระหนักว่า หากต้องการยกระดับโครงสร้างเทคโนโลยีเพื่อก้าวเข้าสู่ยุค AI ให้ประสบความสำเร็จ เราต้องเริ่มจากการผ่าตัดปรับเปลี่ยนวิธีการคุยงาน โครงสร้างการแบ่งฝ่าย และวัฒนธรรมของพนักงานในองค์กรก่อนเป็นด่านแรก
