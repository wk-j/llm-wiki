# Graduated Autonomy (L1–L4) / ระดับอิสระของ AI agent 4 ชั้น

เวลาเราให้ AI agent ไปทำงาน offensive เอง เช่นไปทำ pentest ระบบลูกค้า (ทดสอบช่องโหว่) คำถามแรกเลยคือ "มันรันเองได้แค่ไหน ต้องมีคนคุมใกล้ ๆ มั้ย"

[[owasp-apts|OWASP APTS]] ไม่อยากให้ตอบแบบ autonomous/ไม่ autonomous แต่ให้แบ่งเป็น 4 ชั้น ยิ่งชั้นสูง agent ยิ่งรันเองได้เยอะ ของป้องกันที่ต้องมีก่อนขึ้นชั้นก็เยอะตามไปด้วย

กรอบนี้ใช้มอง coding agent ตัวอื่นก็ได้ ไม่ใช่แค่ pentest [[auto-mode|Auto mode]] ของ [[claude-code|Claude Code]] จริง ๆ ก็คือการเลื่อนจากชั้น 1 ไปชั้น 2 ในงาน dev

## 4 ชั้นคืออะไร

| ชั้น | ชื่อ | คนทำอะไร | Agent ทำเองได้แค่ไหน |
|---|---|---|---|
| **L1** | Assisted | สั่งทีละคำสั่ง | ยิงเทคนิคที่คนบอก ไม่ต่อเอง ไม่เดาขั้นถัดไป |
| **L2** | Supervised | อนุมัติตอนจะข้าม phase | รันหลายเทคนิคต่อกันได้ใน phase เดียว |
| **L3** | Semi-Autonomous | ตั้งขอบไว้ก่อน เข้ามาเฉพาะตอนผิดปกติ | รัน attack chain ยาวจบได้ ถ้ายังอยู่ในขอบ |
| **L4** | Autonomous | รีวิวอาทิตย์ละครั้ง เดือนละครั้ง | คุมหลายเป้าพร้อมกัน ขยาย scope เองได้ |

### Phase ของ pentest คืออะไร

ในมาตรฐาน APTS ใช้ลำดับ phase นี้ตลอด:

```
Recon → Enumeration → Identification → Exploitation → Post-Exploitation → Reporting
```

แปลเร็ว ๆ คือ หาข้อมูลเป้า → ไล่ดูรายละเอียด → หาช่องโหว่ → เจาะ → หลังเจาะทำอะไรต่อ → เขียนรายงาน

L2 รันในช่วงเดียวได้ ถ้าจะข้ามช่วงต้องถามคน ส่วน L3 ข้ามช่วงเองได้เลย

### ทำไมไม่ใช้เปิด/ปิด

ที่แบ่ง 4 ชั้นไม่ใช่เอาสวย เหตุผลจริง ๆ คือความเสี่ยงไม่ได้โตทีละนิด มันโตตามช่วงที่คนไม่ได้ดู ยิ่งช่วงยาวยิ่งเสี่ยง

ยิ่ง agent ต่อ action กันยาวโดยไม่มีคนคั่น โอกาสพลาดก็ยิ่งเยอะ พอแบ่งเป็นชั้น เราจะรู้ว่าตรงไหนต้องมีคนเข้ามา ตรงไหนปล่อยได้ ดีกว่ามาเถียงกันว่า "autonomous" หมายถึงอะไร

## Tier กับ Level ไม่ใช่เรื่องเดียวกัน

ตรงนี้คนชอบสับสน APTS มีสองคำที่ดูคล้ายกันแต่คนละเรื่อง:

- **Tier** บอกว่า platform ผ่าน requirement ชุดไหน มี 3 ชุด (Tier 1, 2, 3) เป็นเรื่องของตัวเครื่องมือ
- **Level** บอกว่าตอนรันงานจริงให้ agent ทำเองมากแค่ไหน มี L1–L4 เป็นเรื่องของงานแต่ละงาน

คร่าว ๆ Tier 1 รันได้แค่ L1 Tier 2 ไปถึง L3 ได้ Tier 3 ถึงจะรัน L4 ได้ แต่ไม่ 1:1 เป๊ะ ๆ

ที่สำคัญคือ platform Tier 3 ก็ยังรันแบบ L1 ได้ถ้าลูกค้าขอ ไม่ใช่ซื้อ Tier 3 แล้วต้องรัน L4 ตลอด

## ขึ้นชั้นทีต้องเพิ่มอะไร

ไม่ใช่กดปุ่มเปลี่ยนชั้นแล้วจบ ก่อนจะขึ้นต้องมีของป้องกันใหม่ติดมาด้วยเสมอ

**จาก L1 ขึ้น L2:**

- ต้องมี gate ให้คนอนุมัติตอน agent จะข้าม phase
- log ต้องละเอียดขึ้น เห็นทีละขั้น
- ตอน agent เสนอ action ต้องให้ operator แก้ได้ก่อนยิง

**จาก L2 ขึ้น L3:**

- กำหนดกลุ่ม action ที่อนุมัติไว้ล่วงหน้าให้ชัด
- มีระบบเฝ้าขอบต่อเนื่อง ไม่ใช่ดูตอนเริ่มตอนเดียว
- มี trigger ให้แจ้งเตือนคนเฉพาะตอนผิดปกติ

**จาก L3 ขึ้น L4:**

- จัดการ campaign หลายเป้าได้โดยคนไม่ต้องเข้ามาระหว่างทาง
- มีกฎว่าค้นเป้าใหม่ยังไง อย่าปล่อยให้มันขยาย scope เอง
- ปุ่ม kill switch ต้องมีคนรับสาย 24/7
- รีวิวเป็นรอบตามกำหนด (อาทิตย์ เดือน ไตรมาส ปี)

L3 กับ L4 ยังต้องทำอีกอย่างเป็นระยะ คือ containment verification (APTS-AL-028) เทสจริงว่า sandbox ยังกั้น agent ได้อยู่ อย่าอ่านเอกสารแล้วเชื่อ

ประโยชน์คือเวลาลูกค้าถามว่า "เราขึ้นไป L3 ได้มั้ย" แทนที่จะตอบใช่/ไม่ใช่ ส่ง checklist ไปให้เขาเช็คว่ายังขาดอะไร ปิดให้ครบแล้วขึ้นได้

## เชื่อมกับของในวิกิ

- [[auto-mode|Auto mode]] ของ Claude Code ทำงานเหมือน L1 → L2 แต่อยู่คนละ layer อันนี้เป็นเรื่องของ harness อันนั้นเป็นเรื่องของ policy
- [[subagent-patterns|Pipeline subagents]] คือวิธีให้ agent หลักไปถึง L2–L3 ได้โดยคนไม่ต้องนั่งดู เอา subagent อื่นมาคอยแทน [[panutat-tejasen|Panutat]] มองมุมเดียวกันนี้แล้วเรียกว่า [[harness-engineering]]
- [[advisor-strategy]] logic คล้ายกับ escalation ของ L3 แต่เป้าหมายคนละอย่าง อันนี้เรื่องประหยัดเงิน ไม่ใช่ความปลอดภัย
- [[delegation-mindset]] ตอนตัดสินใจว่าจะให้ [[claude-opus-4-7|Opus 4.7]] ทำอะไรใน turn เดียว ก็คือเลือก level แบบย่อม ๆ นั่นเอง

## สรุปสั้น

พอมี 4 ชั้น คำถาม "AI agent ปลอดภัยมั้ย" กลายเป็น 3 คำถามที่ตอบได้:

1. ตอนนี้รันชั้นไหน
2. ของป้องกันของชั้นนั้นครบมั้ย
3. ถ้าจะขึ้นชั้น ต้องเพิ่มอะไร

ทำให้ vendor, operator, CISO คุยกันด้วยคำเดียวกันได้ ไม่ต้องมาเถียงว่านิยามของ autonomous คืออะไร
