🪞 ใช้ AI ปิดจุดอ่อน 
จาก Ep. 2 https://www.facebook.com/share/p/1EckyjLCae/ ที่เล่าเรื่อง code review และ testing ที่สำคัญขึ้นเยอะ ผมไม่ชอบ test พอไม่ชอบก็หนี ก็เลยอ่อนประสบการณ์ด้านนี้ พอได้เวลาคืนมาจากการใช้ LLM เขียนโค้ด ก็เลยคิดขึ้นมาได้ว่า เอ ทำไมไม่ให้ Claude มันเขียน test ให้ด้วยล่ะ
ผมเปิด Claude Code ขึ้นมาสั่งไปคำเดียวได้ test มาตามที่เขียนข้างต้น แนืนอนว่ายังไม่เชื่อมัน 100% ก็เลยให้ท่าน Claude ช่วยสรุป test coverage มาให้ ปรากฏว่าเอาตาดูเองแล้ว cover แทบทุกส่วนที่อยาก test เลย
อีกเรื่องที่ผมอ่อนแอมาก ๆ คือ UI/UX ครับ แบบที่เรียกได้ว่าไม่มีหัวคิดด้านนี้เลย หรือถ้าคิดก็ขาดหลักยึดว่าต้องทำอะไรตรงไหนอย่างไร ขาด taste ที่จะมองออกว่า design แบบไหนเหมาะสมกว่ากัน
ก็เรียกท่าน LLM มาช่วยเลยครับ cap หน้าจอเว็บให้ท่าน Gemini ดูแล้วให้ท่านแนะนำ ท่านก็สอนให้รู้จักคำศัพท์ใหม่ ๆ ครับ "Horror Vacui" แปลว่า โรคกลัวที่ว่าง เว็บ cafe-grader ใน Ep. 1 นั้น โดนท่าน Gemini ด่า design ของผม ว่าเป็น  Wall of Text เพราะผมเห็นที่ว่างบนหน้าจอทีไร ก็ใส่อะไรบางอย่างเข้าไปเสมอ ท่านช่วยปรับให้ "เน้นจุดสำคัญ" แทน
ประเด็นแรกที่อยากชวนคิดตามคือ AI ช่วยปิดจุดอ่อนเราได้ครับ อันนี้จะทำได้ต้อง "รู้ว่าตัวเองอ่อนแอตรงไหน"

📉 แต่จุดแข็งก็ไม่ปลอดภัย 
ปิดจุดอ่อนไปแล้ว แล้วจุดแข็งล่ะ?
ผมโชคดีตรงที่ได้รับมอบงานให้หลายด้านครับ ตั้งแต่ Dev, Security, Ops รวมถึง Infrastructure รู้แบบผิว ๆ พอทำได้ กลายเป็น "เป็ด" ก่อนนี้ก็คิดว่ามันเป็นจุดแข็งอันหนึ่ง แต่มันจริงมั้ย? 
ปีที่แล้ว ภาควิชา upgrade ระบบ network เปลี่ยนจาก Cisco ไปใช้ Huawei แทน ผมต้องเป็นคน config ตัว switch กับ router แน่นอนว่าไม่รู้คำสั่งเลยเพราะใช้ Cisco มาตลอด ถามท่าน LLM ท่านก็ช่วยสอนได้หมด ที่น่าตกใจคือพอเอา SFP+ ของ Huawei ไปเสียบ Cisco ตรง ๆ ปรากฏว่า Cisco reject ครับ ตรงนี้ Gemini ช่วยไว้ โดยให้สั่ง `service unsupported-transceiver` เป็นคำสั่งที่ไม่มีใน doc  กด ? บน Cisco ก็ไม่ขึ้น แต่พิมพ์ไปใช้ได้เลย 
ที่ภาคฯมี Proxmox cluster 7 nodes (กราบท่านอาจารย์ มีน Noppakorn Jiravaranun Super TA ที่ถ่ายทอดวิชาให้ผมครับ) ผมให้ท่าน Claude Code ไปอ่าน API doc ของ Proxmox แล้วเขียนเป็น skill ไว้ ตอนนี้จะสร้าง VM ก็แค่สั่งประโยคเดียวจบ
ท่านมีนช่วยติดตั้ง Grafana dashboard, pfSense + wireguard ไว้ให้ด้วย ผมรับช่วงดูแลต่อ งานพวกนี้สั่งท่าน Claude ทีเดียว ได้ json dashboard มาเลย แถมติดตั้ง dcgm-exporter และเขียน prometheus.yml ให้เรียบร้อย
เส้นที่เราเคยขีดว่า 'ตรงนี้ AI ทำไม่ได้' มันขยับไปทุกปีครับ จาก code ไป architecture ไป infra แล้วก็จะขยับต่อไปเรื่อย ๆ
ความรู้พวกนี้ต้องใช้เวลาฝึกพอสมควร แต่ตอนนี้ใครมี AI ก็ทำได้เลย จุดแข็งบน Deployment หรือ Infra กลายเป็นไม่ใช่อีกต่อไปครับ (ย้ำว่าไม่ใช่ไม่ต้องเรียนรู้นะครับ แต่มันง่ายขึ้นเยอะมาก ไม่ต้องไปไล่อ่าน forum อ่าน serverfault หาคำตอบอีกแล้ว)

⚠️ ยิ่งใช้ AI ยิ่งอ่อน
ก่อนจะไปบอกว่า "งั้นก็ outsource ให้ AI หมดสิ" ต้องเล่าอีกด้านครับ
เรื่อง test ข้างบน ทำไปสักพัก ก็เจอว่ามัน test ส่วน authorization ไม่ครบอีกแล้ว แน่นอน เพราะ authorization มัน domain specific มาก ๆ AI ไม่รู้ว่า business logic ของเราต้อง check อะไรบ้าง เราต้องเอ๊ะเอง แล้วไปบอกให้ท่านช่วย "อ่าน code ใหม่ แล้วเขียน test ให้ cover ส่วนนี้ ๆ ๆ ด้วยนะ"
พอยกให้ AI ทำมากขึ้น เราก็ยิ่งไม่ได้ทำเอง ยิ่งไม่ได้ทำเอง ก็ยิ่งไม่พัฒนา "ต่อมเอ๊ะ" ยิ่งไม่มีต่อมเอ๊ะ ก็ยิ่งตรวจงาน AI ไม่ได้ 
ก่อนมี AI เราถูกบังคับให้เรียนรู้ ตอนนี้ AI ทำแทนได้ เราก็เลยไม่ต้องเรียนเลย สบาย แต่ก็แปลว่าเราจะไม่มีทางพัฒนาด้านนั้นขึ้นมาได้เลยเหมือนกัน 
═══════════════════ 
🧭 รู้จักตัวเอง 
ผมคิดว่า key ของ Ep. นี้คือเรื่อง "รู้จักตัวเอง" ครับ
ทุกเรื่องที่เล่ามา ไม่ว่าจะ test, UI, network, Proxmox มันเกิดขึ้นได้เพราะรู้ว่าตัวเองอ่อนตรงไหน พอรู้ก็ใช้ AI ช่วยได้ตรงจุด แต่ที่สำคัญกว่านั้นคือ เราโยนให้ AI แล้วเดินหนีไม่ได้ เราก็ต้องเรียนไปกับมันด้วย ต้องนั่งอ่าน config ที่ AI เขียนให้ ต้องลอง cap หน้าจอไปถาม Gemini แล้วพยายามเข้าใจว่าทำไมมันถึงแนะนำแบบนั้น
การ "รู้จักตัวเอง" ก็คือต้องกล้าก้าวออกไปทำเรื่องที่ไม่ถนัดด้วยครับ ถ้าไม่เคยแตะ testing ก็ไม่มีทางรู้ว่าตัวเองอ่อนเรื่อง testing ถ้าไม่เคยลอง config network ก็ไม่มีทางรู้ว่าตัวเองขาดอะไร ต้องยอมออกจาก comfort zone ถึงจะได้รู้ "รูปร่าง" ของตัวเอง
ตรงนี้ผมกังวลเรื่องผลของ AI ต่อการเรียนรู้ของเรา โดยเฉพาะนิสิตครับ โดยเฉพาะนิสิตครับ นิสิตส่วนใหญ่ยังไม่ได้ทำงานจริงมากพอจะรู้ขอบเขตตัวเอง (จริง ๆ แม้แต่ senior เองก็ต้องปรับไปตลอด) พอมี AI ก็ยิ่งง่ายที่จะเข้าใจผิดว่าตัวเองทำได้ ทั้ง ๆ ที่ AI ทำให้ โดยเฉพาะอย่างยิ่งหากใช้ “ผลงาน” เป็นตัววัด แทนที่จะใช้ “ความเข้าใจ”

❓ แล้วนิสิตควรทำยังไง 
เมื่อก่อนผมเชื่อว่าความเป็น specialist คือ moat ที่ดี จุดอ่อนก็ outsource ไป แต่ตอนนี้ AI มารับ Outsource ไปง่าย ๆ หลายส่วน อ้าวแล้วงั้น moat อยู่ตรงไหน? คนที่รู้กว้างและใช้ AI ปิดจุดอ่อนได้ อาจ deliver งานได้เร็วกว่าคนที่รู้ลึกแค่ด้านเดียว แต่คนที่รู้กว้างแต่ไม่ลึกสักเรื่อง ก็จะไม่มี "ต่อมเอ๊ะ" ในสักด้าน ตรวจงาน AI ไม่ได้เลย
ดังนั้นนิสิตต้อง "เลือก" ครับ เลือกว่าจะลึกเรื่องอะไร กว้างเรื่องอะไร ตรงไหนลงมือทำเอง ตรงไหนให้ AI ช่วย และที่ยากคือตัวเลือกนี้ไม่ได้ตัดสินใจครั้งเดียวจบ เพราะสิ่งที่ควรลึกวันนี้ อีก 6 เดือน AI อาจทำแทนได้หมด ต้องกลับมาประเมินตัวเองใหม่เรื่อย ๆ
สิ่งที่พอจะบอกได้คือ ต้องไม่หยุดเรียนรู้ และต้องกล้าเรียนรู้เรื่องนอก Safe Zone ด้วย เพราะนั่นแหละคือทางเดียวที่จะรู้จักตัวเองจริง ๆ indicator ตัวที่แนะนำก็คือเราได้เรียนรู้อะไรใหม่ ๆ บ้าง โดยเฉพาะเรียนรู้อะไรเกี่ยวกับตัวเองบ้าง
ที่พูดมานี่ ผมก็ทำไม่ได้นะครับ ผมเองก็มี safe zone ชัดเจน พูดแต่เรื่อง hard skill ไม่แตะ soft skill เลย ก็เลยเขียน Ep. นี้ขอความเห็นว่า เราควรจะแนะนำนิสิตของเราอย่างไรดี เสริมจุดแข็งให้แกร่ง หรือใช้ AI ปิดจุดอ่อนให้ครบ หรือจริง ๆ แล้วต้องรู้จักตัวเองก่อนถึงจะตอบได้?
พี่ ๆ น้อง ๆ ท่านใดมีประสบการณ์ด้าน career development มาช่วยเล่าได้จะกราบขอบพระคุณมาก ๆ ครับ
ปล. post นี้ให้ท่าน Claude และ Gemini ช่วยอ่าน ช่วยวิจารณ์ ช่วยด่า ช่วยแนะนำ ช่วยตรวจคำผิด ช่วยเลือก emoji ให้ครับ ถ้าอ่านแล้วสนุก ขอยกความดีให้ LLM ทั้งสองตัวครับ
ปล.2 สัปดาห์หน้าต้องไปช่วยงานแข่งขันคอมพิวเตอร์โอลิมปิกระดับชาติที่ ม. สุรนารี คงได้เจอพี่ ๆ น้อง ๆ อาจารย์ ม. อื่น ๆ อีกหลายท่าน ถ้าได้เรื่องอะไรน่าสนใจจะเอามาเล่าให้ฟังครับ
