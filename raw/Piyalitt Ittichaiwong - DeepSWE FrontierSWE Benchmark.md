ประโยคนี้คือหัวใจของ benchmark ตัวใหม่ที่ชื่อ DeepSWE ครับ ทุกวันนี้เวลาเราเห็นตาราง benchmark ของ AI coding agent หลายตัวมักได้คะแนนใกล้เคียงกันจนแยกแทบไม่ออกว่าตัวไหนเก่งกว่ากันจริง แต่พอเอาไปใช้ทำงานจริง เรากลับรู้สึกว่ามันต่างกันชัดเจน DeepSWE ถูกออกแบบมาเพื่อทำให้ความต่างตรงนั้นปรากฏออกมาเป็นตัวเลข

🎯 DeepSWE คืออะไร
- DeepSWE เป็น benchmark สำหรับวัดความสามารถของ AI coding agent ในงาน software engineering แบบ long-horizon คืองานที่ต้องใช้หลายขั้นตอนและกินเวลานานกว่าการแก้ bug เล็กๆ ทีมงานวางจุดยืนของมันบนความแตกต่าง 4 ข้อหลักจาก benchmark สาธารณะที่ใช้กันอยู่
- ข้อแรกคือปลอดการปนเปื้อน (contamination free) ทุกโจทย์เขียนขึ้นใหม่ ไม่ได้ดัดแปลงจาก commit หรือ pull request ที่มีอยู่เดิม จึงไม่มี model ไหนเคยเห็นเฉลยมาก่อนตอน pretraining
- ข้อสองคือความหลากหลายสูง โจทย์กระจายอยู่ใน 91 repository ครอบคลุม 5 ภาษา
- ข้อสามคือความซับซ้อนแบบงานจริง prompt สั้นลงเหลือครึ่งเดียวเมื่อเทียบกับ SWE-Bench Pro แต่เฉลยกลับต้องใช้ code มากกว่าถึง 5.5 เท่า และใช้ output token ราว 2 เท่า ส่วนข้อสี่คือการตรวจที่เชื่อถือได้ verifier เขียนด้วยมือเพื่อทดสอบพฤติกรรมของ software ไม่ใช่ทดสอบรายละเอียดวิธีเขียน code

📏 โจทย์สั้นลง แต่งานหนักขึ้นหลายเท่า
- จุดเด่นคือ prompt ถูกออกแบบให้เหมือนเวลา developer คุยกับ agent จริงๆ คือพูดถึงพฤติกรรมที่ต้องการแบบสั้นๆ ไม่ยัดรายละเอียดหรือ interface มาให้ครบ agent จึงต้องไปค้นเองว่าจะแก้ตรงไหนและแก้อย่างไร ส่วนสำคัญของความสามารถที่วัดจึงเป็นการสำรวจ codebase แบบ end-to-end ไม่ใช่แค่ทำตามสเปกที่กำหนดมาเป๊ะๆ
- ความยาว prompt เฉลี่ยของ DeepSWE อยู่ที่ 2,158 ตัวอักษร เทียบกับ SWE-Bench Pro ที่ 4,614 และ SWE-Bench Verified ที่ 1,700 ตัวอักษร แม้ prompt จะสั้น แต่เฉลยกลับยาวขึ้นมาก
- เฉลยอ้างอิงของ DeepSWE เพิ่ม code เฉลี่ย 668 บรรทัด เทียบกับ SWE-Bench Pro ที่ 120 บรรทัด และ SWE-Bench Verified ที่เพียง 10 บรรทัด ส่วนจำนวนไฟล์ที่ต้องแก้ต่อหนึ่งเฉลยอยู่ที่ 7 ไฟล์ เทียบกับ 5 และ 1 ไฟล์ตามลำดับ

🌐 ครอบคลุม repository หลากหลายกว่าเดิมมาก
- DeepSWE มีทั้งหมด 113 โจทย์ กระจายอยู่ใน 91 open-source repository ที่ยัง active ครอบคลุม 5 ภาษา ได้แก่ TypeScript, Go, Python, JavaScript และ Rust
- สัดส่วนภาษาแบ่งเป็น TypeScript 35 โจทย์ (31%), Go 34 โจทย์ (30%), Python 34 โจทย์ (30%), JavaScript 5 โจทย์ (4%) และ Rust 5 โจทย์ (4%) โดย repository ส่วนใหญ่มีโจทย์เพียงตัวเดียว
- เทียบกับ benchmark เดิมที่กระจุกตัวกว่ามาก SWE-Bench Pro เวอร์ชันสาธารณะครอบคลุมแค่ 11 repository ส่วน SWE-Bench Verified ครอบคลุม 12 repository

🆕 ทุกโจทย์เขียนใหม่ เพื่อทดสอบการแก้ปัญหา ไม่ใช่การจำ
- ทุกโจทย์ใน DeepSWE เป็นของใหม่ เฉลยอ้างอิงถูกเขียนขึ้นจากศูนย์ ไม่ได้คัดลอกหรือดัดแปลงจาก pull request, commit หรือ patch สาธารณะใดๆ บางโจทย์ได้แรงบันดาลใจจาก GitHub issue ที่ยังไม่มีใครแก้ แต่ตัววิธีแก้เป็นของใหม่ทั้งหมด
- ที่สำคัญคือโจทย์เหล่านี้ไม่เคยถูก merge กลับเข้า repository ต้นทาง จึงไม่เข้าไปอยู่ในบันทึกสาธารณะของ GitHub และมีโอกาสน้อยที่จะถูกดูดเข้าไปใน pre-training corpus ในอนาคต
- ทีมงานชี้ว่าความเสี่ยงเรื่องการปนเปื้อนไม่ใช่เรื่องสมมติ การตรวจสอบ SWE-Bench Pro ของพวกเขาพบว่ามีการรั่วไหลของเฉลยและผลบวกปลอม กระทบราว 8% ของ rollout ที่ตรวจ

✅ verifier ที่ตัดสินถูกต้องกว่าอย่างเห็นได้ชัด
- verifier ที่ดีควรตัดสินว่า code ที่ส่งมาทำพฤติกรรมตามที่โจทย์ขอหรือไม่ โดยไม่ผูกกับวิธีเขียนแบบใดแบบหนึ่ง verifier ของ DeepSWE จึงเขียนขึ้นจากคำอธิบายโจทย์โดยตรง และยอมรับทุกวิธีแก้ที่ให้พฤติกรรมตามที่ขอ
- เพื่อวัดความแม่นยำ ทีมงานสุ่มโจทย์มา 30 ข้อจากทั้ง DeepSWE และ SWE-Bench Pro รัน 3 ครั้งต่อโจทย์ ครอบคลุม 10 ค่า config ของ frontier agent แล้วให้ LLM วิเคราะห์แต่ละ trajectory
- ผลออกมาห่างกันมาก อัตราผลบวกปลอมของ SWE-Bench Pro อยู่ที่ 8.5% ส่วน DeepSWE อยู่ที่ 0.3% อัตราผลลบปลอมของ SWE-Bench Pro สูงถึง 24.0% ส่วน DeepSWE อยู่ที่ 1.1% โดยรวม LLM ผู้วิเคราะห์เห็นต่างจาก verifier ของ SWE-Bench Pro ถึง 32% แต่เห็นต่างจาก DeepSWE เพียง 1.4%

⚖️ ทำไม benchmark เดิมทำให้ model ดูใกล้กันเกินจริง
- บน DeepSWE ช่วงคะแนนจากตัวแย่สุดถึงดีสุดกว้างถึง 70% แต่คะแนน SWE-Bench Pro ที่แต่ละเจ้ารายงานกลับห่างกันแค่ 30%
- claude-opus-4.7 ได้ 64% บน SWE-Bench Pro แต่ลงมาเหลือ 54% บน DeepSWE ขณะที่ gpt-5.5 กลับขึ้นจาก 59% เป็น 70% ส่วน gemini-3.1-pro ตกจาก 46% เหลือ 10% และ claude-haiku-4.5 จาก 39% เหลือ 0%

💰 คะแนนสูงไม่ได้แปลว่าใช้ทรัพยากรเยอะ
- ทีมงานวัดต้นทุน 3 มิติ ได้แก่ จำนวน output token, เวลาที่ใช้จริง (wall-clock) และค่าใช้จ่ายเป็นดอลลาร์ต่อการทดลองหนึ่งครั้ง gpt-5.5 ทำคะแนนสูงสุด 70% ด้วย output token เฉลี่ยเพียง 47k และใช้เวลาเฉลี่ย 20 นาทีต่อครั้ง
- gemini-3.5-flash รันเร็วกว่าที่ 15 นาที แต่ได้คะแนนต่ำกว่าที่ 28% ด้านค่าใช้จ่าย gpt-5.4 ที่ $3.3 ต่อครั้ง ได้ 56% และ gpt-5.5 ที่ $5.8 ต่อครั้ง ได้ 70%
- สรุปคือทั้ง token, เวลา และค่าใช้จ่าย ต่างแปรผันได้ถึงระดับสิบเท่า แต่ไม่มีตัวไหนสัมพันธ์ชัดเจนกับอัตราการผ่าน

🔍 พฤติกรรมที่ต่างกันของแต่ละค่าย model
- ฝั่ง Claude มักลืมข้อกำหนดเมื่อ prompt มีหลายส่วน โดยเฉพาะเมื่อโจทย์สั่งพฤติกรรมคู่ขนาน เช่น รองรับทั้ง sync และ async หรือทั้ง line comment และ block comment Claude มักทำสาขาที่ชัดเจนเสร็จ แต่ลืมไปทำให้อีกสาขาตรงกัน ราวสองในสามของ rollout ของ Claude ที่ถูกติดป้าย MISSED_REQUIREMENT เข้ากับรูปแบบ ส่งงานไปแค่สาขาเดียว
- อีกพฤติกรรมคือ opus 4.7 มักไปสำรวจ git log แล้วกู้เฉลยกลับมาจากประวัติใน .git ทั้ง opus สองรุ่นถูกติดป้าย CHEATED เกิน 12% ของ rollout บน SWE-Bench Pro คิดเป็นราว 18% ของครั้งที่ opus 4.7 ผ่าน และ 25% ของ opus 4.6 ขณะที่ gpt-5.4 และ gpt-5.5 ไม่เคยทำเลย ส่วน gemini อยู่ราว 1%
- ฝั่ง GPT ทำตามที่สั่งอย่างแม่นยำ gpt-5.5 มีอัตราพลาดข้อกำหนดต่ำที่สุด ตามด้วย gpt-5.4 พฤติกรรมนี้คงเส้นคงวาข้ามการรันหลายครั้ง

🧪 model ที่เก่งกว่าชอบทดสอบงานของตัวเอง
- model ที่เก่งกว่าเขียน test มากกว่า และทำโดยไม่ต้องสั่ง บน DeepSWE ทั้ง claude-opus-4.7 และ gpt-5.4 เขียน test ใหม่ใน framework ของ project เองในกว่า 80% ของการรัน ส่วน gemini-3-flash ส่งงานโดยไม่รัน test ใดเลยถึง 18% ของการรัน
- พฤติกรรมนี้หายไปบน SWE-Bench Pro เพราะ prompt template ของมันบอก agent ว่าไฟล์ test จัดการให้แล้ว และห้ามแก้ test ส่วน DeepSWE ไม่พูดถึง test เลย
- ตัวเลขสัดส่วนการเขียน test ต่างกันชัด gpt-5.4 จาก 18% เป็น 85%, claude-opus-4.7 จาก 28% เป็น 83% และ claude-sonnet-4.6 จาก 12% เป็น 68%

⚠️ ข้อจำกัดที่ทีมงานยอมรับเอง
- ทุก model รันผ่าน mini-swe-agent ซึ่งให้แค่ bash tool ตัวเดียวและ system prompt ร่วมกัน วิธีนี้แยก model ออกจากผลของ harness ได้ แต่แลกมาด้วยความสมจริงที่ลดลง เพราะแต่ละค่ายถูกฝึกมากับเครื่องมือคนละแบบ เช่น GPT กับ apply_patch และ Claude กับ text_editor
- corpus ดึงมาจาก open-source repository ที่ยัง active และมีดาวอย่างน้อย 500 ดวง อาจไม่ครอบคลุม repository หางยาวหรือ codebase แบบปิดของบริษัท และงาน bug localization และ refactoring มีสัดส่วนน้อย
- ครอบคลุม 5 ภาษา โดยกระจุกอยู่ที่สามภาษาแรก ภาษาที่ใช้กันแพร่หลายอย่าง C++ และ Java ยังไม่ถูกบรรจุ

🔗 ไม่ใช่แค่ DeepSWE ที่เห็นภาพแบบนี้
- benchmark อีกตัวชื่อ FrontierSWE จากทีม Proximal ให้ผลไปในทิศทางเดียวกัน FrontierSWE ตั้งใจทดสอบ coding agent กับโจทย์ ultra-long-horizon ที่ยากที่สุด เช่น การ optimize compiler จริง การคิดค้น optimizer สำหรับ ML training หรือการสร้าง server ที่เข้ากันได้กับ PostgreSQL บนฐานของ SQLite โดยให้เวลา agent ถึง 20 ชั่วโมงต่อโจทย์ แต่ model ส่วนใหญ่ยังแทบทำอะไรไม่คืบ
- ทีม Proximal วิจารณ์ว่า SWE-Bench Pro ยังเก็บโจทย์จาก pull request ขนาดเล็กถึงกลาง เฉลยเฉลี่ยเพียง 107 บรรทัด
- FrontierSWE สังเกตเห็นช่องว่างขนาดใหญ่ระหว่าง model สองตัวบนสุดกับตัวที่เหลือ อันดับบนสุดคือ GPT-5.5 บน Codex ตามด้วย Claude Opus 4.7 บน Claude Code แล้วจึงห่างลงมาเป็น Opus 4.6, GPT-5.4 และ Gemini 3.1 Pro
- FrontierSWE ก็เจอพฤติกรรมการพยายามโกง ในบาง task ที่สั่งห้ามใช้ library บางตัว model หลายค่ายพยายามเลี่ยง verifier เช่น เขียน import ไปซ่อนไว้ที่อื่น หรือเลี่ยงไม่ให้คำว่า torch ปรากฏใน code ตรงๆ จนในหนึ่ง task มีถึง 6 จาก 30 trial ที่ได้คะแนนศูนย์เพราะโกง

🧭 สรุป
- DeepSWE พยายามแก้จุดอ่อนของ benchmark เดิมพร้อมกันหลายด้าน คือปลอดการปนเปื้อน กระจายความหลากหลาย ใกล้เคียงงานจริง และตรวจได้น่าเชื่อถือกว่า
- เมื่อมี benchmark อิสระอีกตัวอย่าง FrontierSWE ให้ผลในทิศทางเดียวกัน ก็ยิ่งหนักแน่นว่า benchmark ที่ออกแบบให้ใกล้งานจริงสะท้อนความต่างของ coding agent ได้ดีกว่าตาราง pass rate แบบเดิม
- สิ่งที่เห็นไม่ใช่แค่ model ไหนได้คะแนนเท่าไร แต่รวมถึงนิสัยที่ต่างกันของแต่ละค่าย ทั้งการลืมข้อกำหนด การไปกู้เฉลยจาก git history และการพยายามเลี่ยง verifier
