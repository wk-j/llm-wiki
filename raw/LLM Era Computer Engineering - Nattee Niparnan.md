## Ep. 1 — LLM ในห้องเรียน และห้องสอบ

Nattee Niparnan — 10 April at 00:31

🤖วิศวฯคอมจะอยู่อย่างไรในยุคLLMครองเมือง Ep. 1

เทอมนี้ ตอนสอบวิชา Data Structure มีนิสิตคนนึงกดปุ่ม "ขอ hint จาก LLM" จากระบบ grader ของภาควิชา ใน 20 นาทีสุดท้ายก่อนหมดเวลา ท่าน LLM วิเคราะห์ code ที่ส่งมา แล้วถามกลับไปแบบ Socratic method ว่า "แยกแยะ array index กับค่าใน array ให้ออก” แล้วนิสิตแก้จุดนั้น กดส่งใหม่ คะแนนพุ่งจาก 10% ไปเป็น 100%

นี่คือหนึ่งในการทดลองว่าเราจะใช้ LLM ในการเรียนการสอนอย่างไร ในภาควิชาวิศวกรรมคอมพิวเตอร์ จุฬาฯ ครับ

แต่คำถามที่แท้จริงก็ตามชื่อเรื่องเลยครับ และอยากขอความเห็นจากพี่ ๆ น้อง ๆ ด้วยครับ

═══════════════════
🏫LLM ในห้องสอบ

ในวิชา Data และ Algo นั้น เราเน้นการเขียนโปรแกรมมาก ๆ ครับ ศิษย์เก่าหลาย ๆ ท่านก็จะคุ้นเคยกับระบบ cafe-grader https://github.com/nattee/cafe-grader-web ที่ท่าน อ. มะนาว Jittat Fakcharoenphol กับ อ. โป้ง Pramook Khungurn ช่วยกันริเริ่มแล้วผมรับช่วงต่อเป็น maintainer  ปัจจุบัน ระบบนี้คือแบบฝึกหัดเขียนโปรแกรมที่นิสิตส่ง code มาแล้วระบบ test ให้อัตโนมัติว่าผ่านกี่ testcase

โจทย์ใน grader นั้นยากครับ มันไม่ได้ยากที่โจทย์อย่างเดียว แต่ก็ยากเพราะมันเป็น summative assessment ที่ต้องแสดง skill ครบถ้วนในแทบทุกส่วน ตั้งแต่ อ่านโจทย์ เข้าใจ spec เขียน program ที่ compile ผ่าน รับส่ง input/output ได้ถูกต้อง ไปจนถึงหัวใจคือออกแบบและใช้งาน algorithm หรือ data structure ได้เหมาะสม เรียกได้ว่าเป็นยาขมหม้อใหญ่ของนิสิตเลย

ปัญหาคือนิสิตต้องการความช่วยเหลือเยอะมาก เช่น บางคนคิดแนวทางออกแต่เขียนโปรแกรมไม่คล่อง รับ input ผิด ก็ fail ทั้ง ๆ ที่ idea ถูก เรามี Discord forum ให้ถามตอบ แต่คนตอบก็ตอบไม่ไหวหรือตอบไม่ทันใจ แถมพอมี LLM แล้ว ส่งโจทย์ให้ LLM ทำตรง ๆ มันก็ผ่านแทบทุกข้อ ถ้านิสิตถามแบบตรง ๆ ก็ได้เฉลยไปเลยโดยไม่ผ่านกระบวนการเรียนรู้

ทางที่เราเลือกก็คือ ฝัง LLM เข้าไปใน grader เลย แต่ควบคุมวิธีที่มันช่วย พอนิสิตส่ง code แล้วไม่ผ่าน จะมีปุ่มให้กดขอความช่วยเหลือ ระบบก็จะส่ง code ของนิสิตไปพร้อมกับ prompt ที่อาจารย์กำหนดไว้สำหรับโจทย์แต่ละข้อ โดย prompt จะสั่งให้ LLM ใช้ Socratic method คือเน้นถามกลับในจุดที่เป็นประเด็น ไม่ใช่ให้เฉลยตรง ๆ

ระบบนี้เปิดให้ใช้ตอนสอบด้วยครับ แต่มี cost กดถามท่าน LLM ครั้งนึง คะแนนเต็มลดลง 10 แต้ม กดสามครั้งคะแนนเต็มก็เหลือ 70 นิสิตต้องชั่งน้ำหนักเองว่าคุ้มไหม

หลังจากใช้มาเกือบปี feedback จากนิสิตก็คือ บางครั้ง LLM ตอบได้ตรงจุดที่ผิดเป๊ะ แก้แล้วผ่านเลย แต่บางครั้งก็ focus จุดผิดไม่ครบ หรือแก้ตามแล้วก็ยังผิดอยู่ดี ส่วน case hail mary แบบที่เล่าตอนต้นนั้นมีเยอะครับ เห็นชัดเจนใน log

═══════════════════
🔙มาถึงจุดนี้ได้อย่างไร?

ย้อนกลับไปสัก 3 ปี ผมขึ้นเวทีเสวนาเรื่อง "เย่! ไม่ต้องเรียน Python แล้ว (หรือเปล่านะ) มี ChatGPT แล้วนี่นา" (https://www.mycourseville.com/?q=onlinecourse/store/24/course/40677) ที่จัดโดยศูนย์นวัตกรรมการเรียนรู้ จุฬาฯ ประเด็นที่คุยคือ ChatGPT เขียน Python ได้เก่งมากจนทำโจทย์ทุกข้อในวิชา Intro to Com Prog ได้ถูกหมด

สิ่งที่ค้างอยู่ในใจตั้งแต่วันนั้นก็คือ แล้วเราควรจะสอนนิสิตอย่างไร ในโลกที่ coding skill ที่เป็นแกนหลักของหลักสูตรแทบจะถูกแทนที่ได้ 100% ด้วย LLM พูดตรง ๆ ตอนนี้ก็ยังไม่รู้คำตอบครับ แต่สิ่งที่พยายามทำมาตลอดก็คือหาทางทำความเข้าใจมันให้มากขึ้น ลองใช้ให้เยอะ

โชคดีอย่างหนึ่งก็คือภาระหน้าที่ของผมมันคือ "เป็ด" คือต้องทำทุกอย่างตั้งแต่ สอน วิจัย จัดซื้อ software development deployment system admin ไปจนถึงเดินสาย LAN ก็เลยได้ลองใช้ LLM รวมถึง tools ต่าง ๆ ในหลาย ๆ มุม มาถึงวันนี้ หลังจากจ่าย Claude Max + Gemini Ultra มาสองเดือน แล้วเจอว่า dev productivity พุ่งขึ้นสูงปรี๊ด พร้อมกับ "ความกังวล" ที่ยิ่งพุ่งสูงขึ้นตามไปด้วย ก็เลยอยากจะมาเล่าและชวนคุยว่า
แล้ววิศวฯคอมควรอยู่อย่างไรในยุค AI

วันนี้ขอเล่าแค่มุมการเรียนการสอนก่อนนะครับ

═══════════════════
🔧สิ่งที่ปรับไปแล้ว

นอกจากระบบ grader ที่เล่าไป ก็มีอีกหลายแนวทางที่ลองทำครับ

ในวิชา Data Structure กับ Algorithm Design เราใช้ flip classroom อยู่แล้ว มีห้องทำโจทย์ที่นิสิตมาช่วยกันวิเคราะห์ สิ่งที่ปรับเพิ่มก็คือเน้นเอา code ที่ผิดมาให้หาว่าผิดตรงไหนมากขึ้น ทั้งในห้องเรียนและในข้อสอบ แนวคิดก็ตรงไปตรงมาครับ ถ้าเราอยู่ร่วมกับ LLM ก็ต้อง review code เก่ง มี"ต่อมเอ๊ะ" มองให้ออกว่าตรงไหนดีหรือไม่ดี เพราะ LLM มันมักจะทำออกมาถูก หรือไม่ก็ผิดใน edge case ดังนั้น attention to detail กับการวิเคราะห์ก็สำคัญขึ้นเรื่อย ๆ

เรื่อง LLM กับการเรียนรู้ ทางภาคสนับสนุนให้นิสิตใช้ LLM ช่วยเรียนเต็มที่ ทั้ง guided learning mode และ NotebookLM สรุปเอกสาร โดยมีจัด workshop แนะนำการใช้งาน (ขอบคุณ อ. ม๊อค Mock Suwannatat https://www.facebook.com/share/p/18U7sGYkey/ ที่มาช่วยบรรยายให้นิสิตครับ) สำหรับงานที่วัดผลด้วยรายงาน เราลองให้นิสิตส่ง Chat Log มาด้วย เพื่อดูว่าเค้าคุยกับ LLM อย่างไร แล้วให้คะแนนตาม "ความพยายามในการสร้างผลงานที่ดีผ่าน LLM" ถึงยังกำหนด rubric ได้ยาก แต่ก็ดีกว่าให้ส่งแต่ผลลัพธ์

อีกแนวทางที่น่าสนใจคือของ อ. สมชาย ในวิชา Intro to Com Prog ที่ออกข้อสอบให้นิสิตเขียน prompt แทนเขียน code เมื่อเอา prompt ไปสร้าง code แล้ว code ต้องทำงานถูกต้อง (อ่านเพิ่มเติมได้จาก post ของ อ. Ekapol Chuangsuwanich ที่ https://www.facebook.com/share/p/1B3XA4ptYG/)

ส่วนเรื่องสื่อการสอน อ. ชิน Nuttapong Chentanez ก็รับอาสาเขียน skill ให้ Claude Code มาช่วยทั้งเขียนเฉลย review เฉลย และทำ presentation แบบมีภาพเคลื่อนไหวที่ค่อย ๆ อธิบายเฉลยของโจทย์แต่ละข้อ ไว้ อ. ชินทำเสร็จแล้วจะลองขอให้มาเล่าให้ฟังครับ

═══════════════════
❓คำถามที่ยังตอบไม่ได้

ถึงแม้จะลองใช้งาน LLM มาช่วยทุ่นแรงทุ่นเวลาต่าง ๆ ไปได้เยอะ ช่วยให้นิสิตได้ประสบการณ์ที่ดีขึ้น แต่มันไม่ได้ตอบคำถามหลักเลยครับว่า แล้วนิสิตควรจะมี skill อะไรบ้างในยุค LLM เพราะวิชาที่พวกผมสอนนั้น "ยังไม่ได้ปรับ outcome อย่างชัดเจน"

ผมขอยกตัวอย่างหนึ่งคำถามชัด ๆ เช่น “ต้องรู้ว่าเมื่อไรจะใช้ Dijkstra Algorithm (หรือ algo อื่นๆ) ” เป็น outcome ที่สำคัญหรือเปล่า หรือเราเชื่อว่า แค่บอกปัญหาไปให้ท่าน LLM แล้วท่าน LLM ก็จะคิดออกได้เองว่าควรใช้ Dijkstra หรือ algorithm อื่น ๆ ดี ตรงนี้คิดมาเป็นปีแล้วก็คิดไม่ออก คือ ถ้าไม่สอน แล้ววันนึง LLM เลือก algorithm ผิดขึ้นมา นิสิตที่ไม่เคยเรียนจะรู้ได้ยังไงว่ามันผิด นี่ก็วนกลับมาที่เรื่อง "ต่อมเอ๊ะ" อีก คือถ้าไม่เคยเข้าใจลึกพอ ก็จะไม่มีทาง review ได้ แต่จะให้เรียนลึกทุกเรื่องเหมือนเดิมก็ดูจะไม่ make sense ในเมื่อ LLM มันทำแทนได้จริง ๆ

คิดมาเป็นปีแล้วก็ยังคิดไม่ตกครับ แนวทางที่อาจารย์หลายท่านเริ่มเห็นตรงกันคือ soft skill สำคัญขึ้นมาก ทั้งการสื่อสาร การทำงานร่วมกับคนอื่น การตั้งคำถามที่ดี ทางหลักสูตรก็พยายามปรับให้นิสิตได้มีประสบการณ์ทำงานจริงและออกไปนอกมหาวิทยาลัยมากขึ้น

ความฝันระยะยาวอีกอันก็คือ ใช้ LLM customize การเรียนการสอนให้เหมาะกับนิสิตแต่ละคน เพราะต่างคนก็มีวิธีเรียนรู้ที่ต่างกัน แต่จะทำยังไงยังคิดไม่ออก

ถ้าพี่ ๆ น้อง ๆ มีความเห็นหรือไอเดียก็มาคุยกันได้ครับ

ไว้มาต่อ Ep. 2 เรื่องใช้ Claude Code ตลอดทั้ง SDLC กันครับ

## Ep. 2 — Coding Agent กับงานจริง และ taste paradox

Nattee Niparnan — 16h (posted ~2026-04-17)

🤖 วิศวฯคอมจะอยู่อย่างไรในยุค LLM ครองเมือง Ep. 2

จุดเริ่มต้นของ Ep. นี้มาจากงานที่ปรึกษาหนึ่ง ผมรับหน้าที่ทำ PoC Web App เมื่อราว ๆ สองเดือนก่อนครับ ปกติงานสเกลนี้ผมต้องใช้เวลา 1-2 เดือน แต่งานนี้ requirement นิ่งช้ามากจนเหลือเวลาแค่สัปดาห์เดียวก่อนเดดไลน์ ผมเลยต้องไปพึ่ง Coding Agent (Claude Code + Antigravity + Figma Make)

ผลลัพธ์คือโปรเจกต์เสร็จในไม่ถึงสัปดาห์ โดยที่ผมจ่ายค่า Tools ไปแค่ 140 เหรียญ ส่วนผมทำหน้าที่คอย Review และตัดสินใจเป็นหลัก (และโดน AI กัดไปหลายดอก)

พอเจอเรื่องนี้กับตัว ก็ยิ่งห่วงครับว่า การทำงานมันไม่เหมือนเดิมอีกต่อไป แล้วนิสิตวิศวฯคอมในวันนี้ จะอยู่กับ LLM ยังไง? ตามชื่อซีรีส์เลยครับ

═══════════════════
🤖 Agentic AI คืออะไร (ฉบับสั้นมาก)

ถ้าใช้ ChatGPT แบบปกติ เราก็อปโค้ดไปแปะ รัน ดูผล แล้วก็อป error กลับมาถามต่อ งานก็อปแปะพวกนี้ผมเรียกว่า "งานลิง" คือใช้มือทำแบบ Mechanical ไม่ต้องใช้สมอง แต่กินเวลาและแทรกอยู่ในทุก Loop ของการทำงาน

Coding Agent อย่าง Claude Code ต่างออกไปตรงที่มันสร้างไฟล์ รันคำสั่ง อ่านผลลัพธ์ แล้วแก้ไขต่อบนเครื่องเราได้เอง มันวน Loop พวกนี้ได้เองจนกว่าจะสำเร็จ พูดง่ายๆ คือมันเอา "คน" ออกจาก Loop ที่เป็นงานลิง ทำให้เราว่างขึ้น และผลักเราไปอยู่ใน Loop ที่ต้อง "ตัดสินใจ" จริงๆ

═══════════════════
🔄 งานวิศวกรเปลี่ยนไปอย่างไร

หัวใจหนึ่งของงานวิศวกรรมคือการ Compromise ครับ ทุกการตัดสินใจมีข้อดีข้อเสีย เราต้องชั่งน้ำหนักแล้วเลือกทางที่เหมาะกับทรัพยากรและข้อจำกัด เรื่องนี้ไม่เคยเปลี่ยน แต่สิ่งที่เปลี่ยนคือ "ทรัพยากร" ที่เราต้องชั่ง

ตัวอย่างเช่น ตอนเลือก Framework เมื่อก่อนเราจะถามว่า "ทีมถนัดอะไร" หรือ "อะไรเขียนเร็วกว่า" แต่รอบนี้คำถามในหัวผมคือ "Agent เขียน Framework ไหนได้เสถียรกว่า" และ "แบบไหนกิน Token น้อยกว่า" ซึ่งเป็นคำถามที่เมื่อ 2 ปีที่แล้วไม่มีในสมองเลย (Disclosure: แต่สุดท้ายผมแก้ปัญหา Token ไม่พอด้วยการ "เปย์เงินเพิ่ม" ครับ)

พอมี Agent ต้นทุนการเขียนโค้ดลดลงฮวบ สิ่งที่เคยแพงกลายเป็นถูก สิ่งที่เคยไม่แคร์กลายเป็นต้นทุนจริง แถมยังคืนเวลาก้อนใหญ่กลับมาให้เราด้วย คำถามคือเราจะเอาเวลานั้นไปทำอะไร

โปรเจกต์นี้เป็นงานแรกที่ผมใช้ Coding Agent จริงจัง ความรู้สึกเหมือนมี Junior สั่งอะไรได้ตามนั้น ผมเลยมือเติบ Prompt รัวๆ สร้าง feature ไปเรื่อยๆ เอาเวลาที่ได้คืนมาไปใช้กับ "การสร้างเพิ่ม" ทั้งหมด ไม่ได้แบ่งไว้ "คิด" เลย

ซึ่งแน่นอนว่า…. โดน Coding Agent กัด!!! ต่อมเอ๊ะ ไม่ทำงาน แต่ละส่วนย่อยผ่าน Unit Test ดูดี แต่พอทำ Integration Test กลายเป็นว่า Buyer กดลบ Resource ของ Seller ได้ เพราะผมไม่ได้คิดเรื่อง Authorization แบบภาพรวม

พอเราเจอปัญหาและสั่งแก้ Agent ก็แก้ให้ครบทั้งระบบใน Commit เดียวได้สบายครับ แต่ประเด็นคือ "Agent มันไม่รู้หรอกว่าต้องแก้เรื่องนี้ ถ้าเราไม่สั่ง"

เวลาที่ Agent คืนมาให้ ถ้าเอาไปปั่นฟีเจอร์เรื่อยๆ มันก็แค่ "เร็วขึ้น" แต่ถ้าเอามาถอยมองภาพรวม ชั่งน้ำหนัก และทำตัวเป็น Checker ที่คอยออดิทงาน ตรงนั้นแหละครับคือจุดชี้วัดว่างานจะออกมาดีหรือไม่ดี

เพราะการ “โค้ดได้” มันกลายเป็นเรื่องเล็กไปแล้ว ใครยอมควักเงินจ่าย Claude Code ก็ทำได้หมด

❓ แล้วนิสิตควรรู้อะไร?

ขอตั้งข้อสังเกตจากที่เจอมาเฉพาะในมุม Dev ก่อนนะครับ

ทักษะ "ใช้ Agentic AI เป็น" สอนกันไม่ยาก ยัดไว้ใน 1-2 วิชาก็พอ สิ่งที่ยากกว่าคือ "ใช้เมื่อไร ใช้ขนาดไหน และจะ Review ผลของมันอย่างไรให้มีต่อมเอ๊ะ"

แต่ตรงนี้มีปัญหาซ้อนอยู่ครับ คือการจะ review ได้ดี มันต้องเคย "ทำ" มาก่อน ผมจับ bug เรื่อง authorization ได้ เพราะเคยทำมาก่อน รู้ว่าควรจะเอ๊ะตรงไหน test อย่างไร แต่ถ้าเป็นคนที่ไม่เคยเขียน auth เอง เคยแต่สั่ง Agent ให้ทำ จะรู้ได้ยังไงว่าตรงไหนควรเอ๊ะ

นี่คือสิ่งที่ผมเรียกว่าปัญหา "taste" ครับ คือ Agent ทำให้เราข้ามขั้นตอนฝึกมือได้ แต่ขั้นตอนฝึกมือนั่นแหละที่สร้าง judgment ให้เราคุม Agent ได้ มันวนเป็นลูป ต้องเคยทำเอง ถึงจะรู้ว่าอะไรดีหรือไม่ดี ถึงจะ review คนอื่น (หรือ Agent) ได้

อีกสิ่งที่ผมคิดว่าจำเป็นมากขึ้นแบบก้าวกระโดดคือ การตัดสินใจ High-level Design Choice ทำไมต้อง Framework นี้ ทำไม Lib นี้ถึงเหมาะ ตรงนี้ต้องอาศัยความเข้าใจองค์รวมและผลกระทบของระบบ เมื่อก่อนสิ่งเหล่านี้อาจเป็นความรับผิดชอบของ Senior แต่วันนี้ Agent รับงาน Implementationไปแล้ว นิสิตจบใหม่จึงถูกบังคับให้ต้องขยับขึ้นมาคิดภาพใหญ่แทน

อีกเรื่องคือ Core ของความเป็น Engineering จะยิ่งเด่นชัดขึ้น วิชาที่เป็นยาขมอย่าง Data Structure, Algorithm หรือ Discrete Math จะไม่ใช่แค่ทฤษฎีบนกระดาน แต่มันคือพื้นฐานที่จะทำให้คุณเข้าใจระบบและตรวจสอบงานของ AI ได้ ยิ่งมี Agent ยิ่งต้องแม่นเรื่องพวกนี้มากขึ้น ไม่ใช่น้อยลง

คำถามที่ยังตอบไม่ได้คือ แล้วนิสิตต้องฝึกมือ "ลึกแค่ไหน" ถึงจะพอ เพราะถ้าให้ฝึกทุกเรื่องเหมือนเดิมก็ไม่ make sense ในเมื่อ Agent ทำแทนได้จริง แต่ถ้าไม่ฝึกเลย ก็ไม่มีทางมี taste ที่จะคุม Agent ได้เลย

สุดท้าย "วันนี้" Agent ยังไม่เก่งเรื่องมองภาพรวม แต่ก็ไม่มีใครรู้ว่า “พรุ่งนี้” model เทพ ที่ทำแทนเราได้ 100% จะมาหรือไม่ (ตามอ่าน อ. อ๋า Piyalitt Ittichaiwong บ่อย ๆ นะครับ)

ยิ่งเป็นเหตุผลที่ต้องรีบสร้าง skill พวกนี้ตอนที่มันยังมีค่าอยู่

ใครทำงานจริงแล้วเจออะไรที่เปลี่ยนไปเพราะ Agentic AI มาแชร์กันได้นะครับ ผมต้องกราบขอความรู้จากคนทำงานจริงอีกเยอะครับ

ปล. ที่ผมบอกว่าสอนใช้ Agentic AI ไม่ยาก เอาเข้าจริงแอบมีอุปสรรคเรื่องค่าใช้จ่ายรายเดือนของ Tools ระดับ Top อยู่บ้าง ตรงนี้ต้องกราบขอบพระคุณ อ. Ekapol Chuangsuwanich และ อ. Dittaya Wanvarie ที่กำลังช่วยประสานงานกับหลายๆ เจ้าเพื่อแก้ปัญหานี้ให้นิสิตเราอยู่ครับ
