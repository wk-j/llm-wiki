Senestia is a health tech company which aim to increase healthcare services’ accessibility so people can have a better quality of life.

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*153B92OemM_akvZBy1oWvw.png)

เราดีใจที่เดี๋ยวนี้เรามี AI (LLMs) มาช่วยเขียนโค้ด จากงานที่เคยใช้เวลาหลายวัน เหลือเพียงไม่กี่นาที แต่พอหันไปมองภาพรวมในมุมของลูกค้า หรือตัวเลขการส่งมอบงาน (Lead Time) กลับพบว่า… **“มันแทบไม่เร็วขึ้นเลย”**

ทำไมถึงเป็นแบบนั้น? คำตอบไม่ได้อยู่ที่ว่าเราเขียนโค้ดเร็วแค่ไหน แล้วอยู่ที่อะไร?

## 1\. กับดักของการเร่งสปีดผิดจุด

ในเชิงทฤษฎีการจัดการ (Theory of Constraints) กล่าวไว้ว่า *“* ***ความเร็วของกระบวนการทั้งหมด จะเท่ากับความเร็วของส่วนที่ช้าที่สุด (Bottleneck) เท่านั้น*** *”*

ลองนึกภาพ Pipeline การทำงานของเรา:

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*4zQt56L8Ckjm6gpoJ4tCCA.png)

หากเราเร่งสปีดที่จุด **Coding** อย่างเดียวจนเร็วเป็นจรวด แต่ขั้นตอนนี้เป็นเพียง 10% ของเวลาทั้งหมด ส่วนที่เหลืออีก 90% คือการรอคอย ผลลัพธ์ที่ได้คือ งานที่เสร็จจาก Code ไปกองรวมกันอยู่ที่ขั้นตอนถัดไปแทน

## 2\. การรอ คืออุปสรรคที่ใหญ่ที่สุดของ Flow

เหตุการณ์สมมติที่เจอได้ทั่วไป:

- Code เสร็จใน 30 นาที… แต่รอ Review Pull Request อีก 3 วัน
- Code เสร็จใน 30 นาที… แต่รอหยิบไป Test อีก 3 วัน
- เจอ Edge case… รอพิจารณาแนวทาง อีก 3 วันจะสรุป

> สุดท้าย งาน 30 นาทีนี้ใช้เวลาเดินทางถึงมือลูกค้าอย่างน้อย 3 วัน การที่เราเขียนโค้ดเร็วขึ้น 10 เท่า จึงแทบไม่มีความหมายเลย

## 3\. “If you can’t see it, you can’t fix it”

วิธีแก้ไขคือการ **“ทำให้คอขวดมองเห็นได้”** พวกเราลองเริ่มเก็บข้อมูลจากบอร์ดงานในสปริ้นเพื่อหาว่างานไป “คอขวด” อยู่ที่ไหน:

- **เดือนแรก:** พบว่าคอขวดอยู่ที่ `In Progress` แต่เอ๊ะ!! `Ready to Test` (รอหยิบงานไปเทส) เฉลี่ยสูงถึง **2.8 วัน** ต้องมีอะไรผิดปกติแน่ๆ โฟกัสตรงนี้ก่อนละกัน
![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*duQbraYbcaQI0aPS-h6IPg.png)

- **เดือนถัดมา:** เราโฟกัสการแก้ปัญหาที่จุดเดิม จนลดเหลือ **1.3 วัน** และ **1.1 วัน** ตามลำดับ
- **ปัจจุบัน:** เราสามารถคุมเวลา `Ready to Test` ให้เหลือเพียง **16 ชั่วโมง**
![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*YOxQkSiAAfITJub2vjuDgg.png)

ใช่แล้ว ผมเลือกแคปจอตอนสวยๆ

> Note: จากรูป stage In Progress ไม่ได้แยก Implementation กับ Code Review ออกจากกัน เลยทำให้แยกสองส่วนนี้ไม่ได้

## 4\. นอกจากหยิบงานไปเทสเร็วขึ้นแล้ว เราทำอะไรไปอีกบ้าง?

เมื่อเราเห็นตัวเลขชัดๆ ว่างานไป “นอนรอ” อยู่ที่ไหน นี่คือ 4 สิ่งที่เราลงมือทำเพื่อแก้ปัญหาที่ต้นเหตุ:

- **Break Down Tasks:** งานที่มักจะไม่เสร็จตามแผน คืองานชิ้นใหญ่ที่ซ่อนความไม่แน่นอน (Unknowns) ไว้เยอะ เราสนับสนุนให้ Break down งานจนเล็กพอที่จะจบได้ใน 1–2 วัน ซึ่งช่วยให้การ Review และ Testing ทำได้รวดเร็วตามไปด้วย
- **Shift-Left Testing:** เราแก้ปัญหาการรอหยิบงานไปเทส (Ready to Test) โดยการเตรียม Test Case ให้เสร็จตั้งแต่ก่อนเริ่ม Sprint Planning วิธีนี้ช่วยให้ทุกคนเห็นภาพตรงกันว่า “เสร็จ” คืออะไร และลดความล่าช้าจากการรอคนมาเขียนเทสเคสทีหลัง
- **Self-Testing:** แทนที่จะส่งงานข้ามไปข้ามมา เราให้คนที่ Implement เป็นคนรันเทสตาม Test Case ที่เตรียมไว้ด้วยตัวเองทันทีที่เขียนโค้ดเสร็จ ผลลัพธ์คือเราได้ Feedback ที่เร็วขึ้นมาก ลดอัตราการ Reject งานจากการเทส และลดการงานนอนกองค้างใน Stage รอเทสนานหลายวัน
- **Limit WIP:** เราใช้กฎ **“Stop Starting, Start Finishing”** โดยจำกัดจำนวนงานในมือไม่ให้มากเกินไป ถ้างานเก่ายังติดอยู่ที่ Code Review หรือ Testing เราจะไม่เริ่มงานใหม่ แต่จะเข้าไปช่วยเพื่อนเคลียร์งานเก่าใน Pipeline ให้จบก่อน เพื่อสร้างจังหวะ (Pace) เดียวกันให้เกิดขึ้นทั้ง Value Stream

## สรุปส่งท้าย

AI อาจช่วยย่นเวลา Coding ให้เราได้ แต่การส่งมอบงานได้ไวไม่ได้เกิดจากการแข่งกัน “เริ่มงานใหม่” ให้เร็วที่สุด หรือทำหลายอย่างพร้อมกัน แต่เกิดจากการจัดการสายพานให้ไม่มีจุดสะดุด ทลายคอขวดที่ซ่อนอยู่ และเปลี่ยนโฟกัสมาที่การ “ปิดงานเก่า” ให้จบทีละชิ้น เพื่อขจัดจุดที่งานต้องไปนอนรอให้หมดไป

ท่องเอาไว้: *“งานจะเสร็จเร็วเท่ากับส่วนที่ช้าที่สุดเสมอ”* วันนี้… ลองเปิดบอร์ดดูนะครับว่า คอขวดของทีมคุณซ่อนอยู่ที่ไหน?
